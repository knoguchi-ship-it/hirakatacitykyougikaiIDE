# 00_OPERATING_MODEL

## 位置づけ
この文書は、AI駆動開発における運用モデルを定義する。  
目的は、AIが暴走せず、人間のレビュー効率を高め、知識をプロジェクト資産として残すことである。

## 基本方針
- AIは自律的に作業してよいが、**無制限の裁量**は持たない。
- AIの「学習」はモデル内部の恒久学習ではなく、**プロジェクト文書の更新**として外部化する。
- 常設ルールは短く保ち、詳細は playbook / docs / skills に分割する。
- まず既存システムを尊重し、新規設計は必要な時だけ行う。

## ルールの優先順位
1. ユーザー依頼 / Issue / PR / 受け入れ条件
2. 安全・法務・セキュリティ要件
3. リポジトリ常設指示 (`AGENTS.md` / `CLAUDE.md` / Copilot instructions)
4. この `docs/AI_RULES/`
5. コード、テスト、設定、README などの実装事実
6. 一次ソース（公式Docs、公式Issue、RFC、ベンダー公式）
7. 二次ソース（ブログ、記事、動画、Q&A）

## 変更原則
- まずは最小の安全な差分で解決する。
- 全面更新は例外であり、合理的理由がある場合のみ行う。
- 設計を変える場合は、変更理由、影響範囲、移行、ロールバックを説明する。

## Web検索の使い方
次の情報は一次ソース確認を原則とする。
- 新規ライブラリやフレームワークの導入
- バージョン差異、破壊的変更、非推奨化
- 外部API仕様
- セキュリティ修正、CVE、既知不具合
- インフラ、課金、法規、運用ポリシー
- エラーが内部知識だけでは確信できない場合

次の情報は、まずリポジトリ内事実を優先する。
- コーディング規約
- ディレクトリ構造
- 既存テストの流儀
- 命名規約
- 依存導入済みの設計パターン

## 完了条件
以下を満たして初めて「完了」とみなす。
- 依頼された変更が実装されている
- 主要リスクが説明されている
- 実施した検証が報告されている
- 未実施検証と残リスクが明記されている
- 必要な文書更新が終わっている
