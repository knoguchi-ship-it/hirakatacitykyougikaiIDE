# 00_OPERATING_MODEL

更新日: 2026-04-05

## 目的
- AI を「自動化の近道」ではなく、統制された開発実務の一部として扱う。
- 最小差分で前進しつつ、評価、説明、承認、記録を欠かさない運用モデルを定義する。
- 会話、コード、証跡、正本ドキュメントの整合を保つ。

## 外部ベースライン
2026-04-05 時点では、次の考え方を最低基準として取り込む。
- NIST AI RMF 1.0: `Govern / Map / Measure / Manage` を AI 作業にも適用する。
- NIST SSDF 1.1 / SSDF-AI: secure-by-design、変更管理、検証、サプライチェーン管理を明示する。
- OWASP GenAI: prompt injection、データ漏えい、不適切な出力処理、過剰権限、サプライチェーンを先回りで抑える。
- OpenAI / Anthropic 公式 prompting guidance: 指示の明確化、構造化、例示、バージョン管理、eval を標準化する。

## 基本姿勢
- まず読む。推測で壊さない。
- まず現在地を確認する。現況、正本、既存差分、制約、権限境界を把握してから動く。
- 局所修正を優先する。全面更新は安全性が明確に上回る場合だけ行う。
- AI の出力は提案であって事実ではない。根拠確認と検証を前提に扱う。
- 説明責任を持つ。何を変え、なぜ変え、どう確かめたかを残す。

## 参照優先順位
1. ユーザー依頼、Issue、PR、受け入れ条件
2. 安全、法務、権限、運用上の制約
3. リポジトリ直下の運用入口 (`AGENTS.md` / `CLAUDE.md` など)
4. `GLOBAL_GROUND_RULES/docs/AI_RULES/`
5. プロジェクト正本 (`HANDOVER.md`, `docs/*`)
6. コード、設定、テスト、README
7. 必要時のみ一次ソースの外部情報

## 標準ループ
1. Govern
   - 目的、責任者、承認要否、境界条件を明確にする。
2. Explore
   - 関連ファイル、設定、ドキュメント、既存差分、既知障害を確認する。
3. Plan
   - 問題、仮説、変更方針、評価方法、ロールバック方法を短く整理する。
4. Execute
   - 既存パターンに合わせて最小差分で変更する。
5. Measure
   - typecheck、build、テスト、実機確認、ログ確認などで結果を測る。
6. Manage
   - 残リスク、未検証、承認待ち、次アクションを明示する。
7. Document
   - 正本、handover、task、evidence を同ターンで更新する。

## Web の使い方
- 法務、規制、セキュリティ、課金、外部サービス仕様、最新標準、脆弱性、ベンダー仕様変更は一次ソース確認を原則とする。
- 古い記憶で断定しない。確認日と出典を残す。
- 外部情報が案件ルールと衝突する場合は、案件ルールを優先し、差分と理由を記録する。

## AI 出力の扱い
- AI の提案は、コード・設定・ドキュメント・実環境の事実で裏取りする。
- prompt や system 指示はソースコード同様に version 管理対象として扱う。
- 例示、出力形式、成功条件、禁止事項を曖昧にしない。
- 評価なしの prompt 更新、本番挙動変更、運用変更は完了扱いにしない。

## 検証と完了条件
- 変更箇所の静的確認を必ず行う。
- 影響範囲に応じて lint / test / build / typecheck / E2E / 実ブラウザ確認を実施する。
- 本番、認証、権限、DB、デプロイは実運用経路での確認を省略しない。
- 未検証、残課題、前提条件、承認待ちがある場合は明記する。

## 承認の考え方
- 本番 deploy、DB 更新、権限変更、外部送信、不可逆操作、対外影響のある操作は人間承認を前提とする。
- 承認が必要な操作では、実施内容、理由、影響範囲、代替案、戻し方を先に示す。
- 迷う操作は進めず、承認要否を明確にしてから実行する。

## 記録の原則
- 状態の正本は `HANDOVER.md` と案件の `docs/*` に置く。
- task 途中経過は task 文書に残し、完了時は正本へ転記する。
- 相対日付ではなく絶対日付を使う。
- 文字化け、参照切れ、版ずれ、古い固定値を見つけたらその場で直す。
