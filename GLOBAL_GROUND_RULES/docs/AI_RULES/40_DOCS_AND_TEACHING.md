# 40_DOCS_AND_TEACHING

更新日: 2026-04-05

## 基本姿勢
ドキュメントはコードの副産物ではなく、システム理解のための一次資産である。  
AIは「実装したら終わり」ではなく、人間が理解・保守・監査できる状態まで整える。

## ドキュメント原則
- コードと矛盾する説明を書かない
- 図はコード、型、状態遷移と一致させる
- 大事な判断は背景と代替案まで残す
- 「なぜ」を残し、「何をしたか」だけで終わらせない
- 運用固定値、承認境界、復旧手順は曖昧語ではなく明示値で書く
- AI への指示や prompt 変更も、人間が読める形で理由を残す

## 必須文書
- `docs/01_ARCHITECTURE/`
  - システム構成
  - データモデル
  - 状態遷移
  - 外部依存
  - 境界条件
- `docs/03_ADR/`
  - 技術判断の記録
- `docs/04_KNOWN_ERRORS.md`
  - 既知エラーと復旧法

## 追加で重視する文書
- `HANDOVER.md`
  - 現在状態、固定 version、次担当者の最初の一手
- 開発プレイブック / runbook
  - 手順、完了条件、証跡、承認境界
- task 文書
  - 長期作業の途中経過、停止点、未了事項

## 視覚化ルール
以下は視覚化を優先する。
- DB構造
- 状態遷移
- 非同期処理フロー
- 認証 / 認可フロー
- 外部連携
- エラー復旧手順
- 監視 / アラートから復旧までの流れ

推奨表現:
- Mermaid
- 単一HTML
- SVG
- シーケンス図
- フローチャート
- 比較表

## 人間向け説明を必須化する条件
- 新技術導入
- 大きな設計変更
- 高リスクな障害対応
- 運用手順の変更
- オンボーディングで詰まりやすい領域
- prompt / policy / tool 連携の変更
- agent の権限境界や停止点を変える変更

## 良い引継ぎ文書の条件
- 今日の状態が先に書いてある
- 絶対日付、version、deployment、PASS / FAIL がある
- 次担当者が最初の 10 分で行動できる
- 未了事項とブロッカーが区別されている
- 参照先が現行の正本に揃っている

## 説明テンプレート
- 背景
- 何が問題か
- なぜその設計か
- 代替案は何か
- どんなトレードオフがあるか
- どこを見れば理解できるか
- 障害時はどう戻すか
