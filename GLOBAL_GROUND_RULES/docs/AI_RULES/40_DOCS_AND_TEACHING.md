# 40_DOCS_AND_TEACHING

## 基本姿勢
ドキュメントはコードの副産物ではなく、システム理解のための一次資産である。  
AIは「実装したら終わり」ではなく、人間が理解・保守・監査できる状態まで整える。

## ドキュメント原則
- コードと矛盾する説明を書かない
- 図はコード、型、状態遷移と一致させる
- 大事な判断は背景と代替案まで残す
- 「なぜ」を残し、「何をしたか」だけで終わらせない

## 必須文書
- `docs/01_ARCHITECTURE/`
  - システム構成
  - データモデル

## 視覚化ルール
以下は視覚化を優先する。
- DB構造
- 状態遷移
- 非同期処理フロー
- 認証 / 認可フロー
- 外部連携
- エラー復旧手順
- 監視 / アラートから復旧までの流れ

推奨表現:
- Mermaid
- 単一HTML
- SVG
- シーケンス図
- フローチャート
- 比較表

## 人間向け説明を必須化する条件
- 新技術導入
- 大きな設計変更
- 高リスクな障害対応
- 運用手順の変更
- オンボーディングで詰まりやすい領域

## 説明テンプレート
- 背景
- 何が問題か
- なぜその設計か
- 代替案は何か
- どんなトレードオフがあるか
- どこを見れば理解できるか
- 障害時はどう戻すか
