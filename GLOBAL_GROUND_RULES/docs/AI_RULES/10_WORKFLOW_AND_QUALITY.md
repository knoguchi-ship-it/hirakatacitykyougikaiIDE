# 10_WORKFLOW_AND_QUALITY

## 標準ワークフロー
1. **Explore**
   - 関連ファイル、テスト、設定、README を読む
   - 既存の実装パターン、制約、依存関係を把握する
   - 無関係なファイルは読まない

2. **Plan**
   - 問題、原因仮説、変更方針、検証方法を短く整理する
   - 複雑タスクでは、先に人間へ計画を共有する
   - 大きい作業は小さなサブタスクへ分割する

3. **Search**
   - 必要時のみ、一次ソースで最新情報を確認する
   - 複数の候補がある場合は、標準性、成熟度、保守性で比較する

4. **Execute**
   - 差分修正を原則とする
   - 変更は一貫性を保つ
   - 既存の命名・責務分割・エラーハンドリングに合わせる

5. **Verify**
   - 最低限、影響箇所のテストを行う
   - 可能であれば lint / test / build / typecheck / E2E を行う
   - 変更差分に対して自己レビューを行う

6. **Document**
   - 技術判断は ADR
   - 再発エラーは KNOWN_ERRORS
   - 学習価値が高い内容は人間向け説明資料に落とす

## 品質基準
- 正しさ
- 読みやすさ
- 変更容易性
- 監視・デバッグ容易性
- セキュリティ
- テスト可能性
- 既存資産との整合性

## 全面更新を許可する条件
以下のいずれかに該当する場合のみ、全面更新を許可する。
- 生成ファイルやロックファイルの再生成
- 機械整形により差分が全面化する場合
- API / schema / config の構造移行
- 壊れたファイルを安全に復旧する必要がある場合
- 差分修正よりも全面更新の方が明確に安全と判断できる場合

その際は必ず:
- 理由
- 影響範囲
- ロールバック方法
- 検証結果
を明記する。

## 検証基準
### 必須
- 変更箇所の静的確認
- 失敗パスの考慮
- 影響範囲の確認

### 可能なら実施
- lint
- unit test
- integration test
- build
- typecheck
- security scan
- representative scenario check

## 変更報告テンプレート
- 何を変えたか
- なぜ変えたか
- どこに影響するか
- どう検証したか
- 何が未検証か
- 次に人間が見るべき点
