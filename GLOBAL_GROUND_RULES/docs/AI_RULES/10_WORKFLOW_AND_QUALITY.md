# 10_WORKFLOW_AND_QUALITY

更新日: 2026-04-09

## 標準ワークフロー
1. **Explore**
   - 関連ファイル、テスト、設定、README、既知障害を読む
   - 既存実装パターン、制約、依存関係、権限境界を把握する
   - 無関係なファイルは読まない

2. **Plan**
   - 問題、原因仮説、変更方針、検証方法、ロールバック方法を短く整理する
   - 複雑タスクでは、実装前に計画と完了条件を共有する
   - 大きい作業は小さなサブタスクへ分割する

3. **Search**
   - 必要時のみ一次ソースで最新情報を確認する
   - 複数候補がある場合は、標準性、成熟度、保守性、導入コストで比較する
   - 調べた内容は日付と出典を残す

4. **Execute**
   - 差分修正を原則とする
   - 変更は一貫性を保ち、既存の命名・責務分割・エラーハンドリングに合わせる
   - prompt、手順、設定、コードを別物として扱わず、一つの変更セットとして整合させる
   - **push / deploy 前に必ず `git diff` で作業ツリー全体を確認し、自分の変更以外の未コミット変更が存在する場合はその影響範囲を評価してから進む。**
     - 他者・前セッションの変更が混在していないか
     - 混在する変更が今回の push に含まれても問題がないか
     - 問題がある場合は push 範囲をファイル単位に限定するか、ユーザーに確認する

5. **Verify**
   - 最低限、影響箇所のテストを行う
   - 可能な範囲で lint / test / build / typecheck / E2E / runtime check を行う
   - 差分に対して自己レビューし、失敗パスと権限境界を再確認する

6. **Document**
   - 技術判断は ADR
   - 再発エラーは KNOWN_ERRORS
   - 運用変更、学習価値の高い変更、引継ぎが必要な変更は人間向け資料へ落とす

## 品質基準
- 正しさ
- 再現性
- 読みやすさ
- 変更容易性
- 監視・デバッグ容易性
- セキュリティ
- テスト可能性
- 既存資産との整合性

## AI 時代の追加品質基準
- **Instruction clarity**: 指示、制約、出力形式、禁止事項が曖昧でない
- **Eval readiness**: 変更前後を比較できる確認手順がある
- **Traceability**: どの文書、どの証跡、どの runtime 確認に基づくか辿れる
- **Least surprise**: 既存 UI / API / 運用を不必要に変えない
- **Tool discipline**: ツール呼び出しや自動操作の境界が明確

## Prompt / Rule / 手順の変更ルール
- prompt、system 指示、運用ルールもコードと同等にレビュー対象とする
- 書き換え時は、成功条件、禁止事項、例、検証方法を合わせて見直す
- 変更後は、少なくとも代表ケースで期待どおりの挙動を確認する
- 可能なら変更前後比較の evidence を残す

## 全面更新を許可する条件
以下のいずれかに該当する場合のみ、全面更新を許可する。
- 生成ファイルやロックファイルの再生成
- 機械整形により差分が全面化する場合
- API / schema / config / 運用ルールの構造移行
- 壊れたファイルを安全に復旧する必要がある場合
- 差分修正より全面更新の方が明確に安全で、レビュー可能性も保てる場合

その際は必ず次を明記する。
- 理由
- 影響範囲
- ロールバック方法
- 検証結果

## 検証基準
### 必須
- 変更箇所の静的確認
- 失敗パスの考慮
- 影響範囲の確認（自分の変更だけでなく、作業ツリー全体の未コミット変更を含む）
- 関連文書との整合確認
- push / deploy 前の `git diff` による全差分確認（他セッションや前工程の変更の混在検出）

### 可能なら実施
- lint
- unit test
- integration test
- build
- typecheck
- security scan
- representative scenario check
- runtime health check

## 変更報告テンプレート
- 何を変えたか
- なぜ変えたか
- どこに影響するか
- どう検証したか
- 何が未検証か
- 次に人間が見るべき点
