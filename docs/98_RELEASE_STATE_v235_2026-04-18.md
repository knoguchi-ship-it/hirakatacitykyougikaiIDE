## v232 — MASTER専用 物理削除機能（データ管理コンソール）

### 目的
誤って登録されたデータを MASTER 権限者が物理削除できる UI を追加。

### 実装内容
- **GAS**: `searchMembersForDelete_` / `previewDeleteMember_` / `executeDeleteMember_` / `getDeleteLogs_` / `addDeleteLogSheet` を追加
- **GAS**: `T_削除ログ` シートを新設（削除前スナップショット JSON を保存）
- **GAS**: cascade 削除順: T_ログイン履歴 → T_年会費更新履歴 → T_研修申込 → T_認証アカウント → T_年会費納入履歴 → T_管理者Googleホワイトリスト → T_事業所職員 → T_会員
- **Frontend**: `MemberDeleteConsole.tsx` 新規作成（検索→プレビュー→「物理削除」テキスト入力→実行 の 3 ステップ確認フロー）
- **Frontend**: `Sidebar.tsx` に MASTER 専用「データ管理コンソール」メニューを追加
- **Frontend**: `App.tsx` に `'member-delete'` ビューを追加
- **初回作業**: `npx clasp run addDeleteLogSheet` で T_削除ログ シートを作成すること

### 制約
- 1回の操作で最大 10 件まで削除可能。回数制限なし。
- MASTER 権限専用（API・UI 両層で制御）

## v233 — 事業所職員重複バグ修正

### 問題
個人会員⇔事業所会員を複数回往復すると、T_事業所職員 に同一人物の ENROLLED レコードが重複して作成されていた。

### 根本原因（3点）
1. `convertIndividualToStaff_`: 同一介護支援専門員番号×同一事業所に ENROLLED が既存かどうかチェックせず `appendRowsByHeaders_` で常に新規追加
2. `assertSingleActiveAffiliationByCareManager_`: staff+staff 重複を検出しない（individual+staff の共存しか見ていなかった）
3. `convertStaffToIndividual_`: `today`/`now` を使う前（代表者チェック Step3）に変数が宣言されていない（JS hoisting で undefined 参照）

### 修正
- `convertIndividualToStaff_` Step 3.5: 同一 CM 番号×同一事業所の ENROLLED 既存チェックを追加（冪等性ガード）
- `assertSingleActiveAffiliationByCareManager_`: `staffRows.length > 1` の場合もエラーを投げるよう拡張
- `convertStaffToIndividual_`: `var now` / `var today` の宣言を Step 3 より前に移動
- `repairDuplicateStaffRecords_()`: 既存の重複 ENROLLED レコードを最新1件以外 LEFT+deleted に修復（MASTER専用 API・UI）

## v234 — 研修申込 申込者ID不整合修正

### 問題
`migrateTrainingApplications_` が `会員ID`・`職員ID` を更新する際に `申込者ID` を更新しなかったため、ロール変換後に `isTrainingApplicationRowValid_` が「会員IDと申込者IDが不一致」と判定し、その人の研修申込が全件 invisible になっていた。

### 修正
- `migrateTrainingApplications_`: `申込者ID = newMemberId` を同期する1行を追加。`削除フラグ=true` のレコードは変更しない。
- `repairTrainingApplicationApplicantIds_()`: 既存の不整合レコード（`申込者ID ≠ 会員ID`、`申込者区分コード=MEMBER`）を一括修復。`会員ID` が T_会員 に存在しない場合はスキップ（MASTER専用 API・UI）。

### 注意
スキップ件数が返った場合は T_研修申込 の該当レコードを手動確認すること。

## v235 — セッションアンカーを loginId に変更（ロール変換後のマイページ表示不整合修正）

### 問題
フロントエンドが `memberId`（ロール変換のたびに変わる値）をセッション識別子として保持していた。管理者がロール変換を行った後、会員がマイページを開くと変換前の `memberId` でポータルデータが取得され、WITHDRAWN の個人会員画面が表示されていた。

### 根本原因（モデリング欠陥）
`memberId` はビジネスオブジェクト識別子（ロールごとに変わる）であり、セッションアンカー（不変）として使うべきではない。`loginId` は変換しても不変。

### OWASP 原則
> "Authentication always resolves to person first, then checks permissions."

### 修正
**バックエンド** `getMemberPortalData_`:
- `loginId` を受け取り、T_認証アカウントから **現在の有効な** `memberId`/`staffId` を解決してポータルデータを返す
- レスポンスに `resolvedMemberId`/`resolvedStaffId` を追加
- 後方互換: `memberId` 直接指定も引き続き動作（管理者代理参照用）

**フロントエンド** `App.tsx`:
- `authenticatedContext` に `loginId` フィールドを追加
- `applyAuthContext` が `auth.loginId` を自動保存
- `loadMemberPortalData`: 引数を `loginId` に変更。バックエンドが返した `resolvedMemberId`/`resolvedStaffId` が現在と異なる場合、`authenticatedContext` と `selectedIdentityId` を自動補正

**フロントエンド** `api.ts`:
- `getMemberPortalData` のシグネチャを `loginId: string` に変更、`resolvedMemberId`/`resolvedStaffId` を返す型に更新

### 動作フロー（修正後）
```
個人M1でログイン → authenticatedContext = { loginId: "xxx", memberId: M1 }
管理者が変換   → T_認証アカウント: memberId = OfficeX, staffId = S1 に更新
会員がマイページへ → loadMemberPortalData("xxx")
              → GAS: loginId照合 → 現在 memberId=OfficeX, staffId=S1
              → OfficeX の事業所会員マイページを返す ✓
              → フロント: resolvedMemberId=OfficeX で authenticatedContext 自動補正 ✓
```
