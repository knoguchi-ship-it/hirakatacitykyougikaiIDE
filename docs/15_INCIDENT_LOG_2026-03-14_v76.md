# 障害記録（2026-03-14 / v76）

更新日: 2026-03-14  
対象バージョン: `@75`（再現） / `@76`（修正）

## 1. 概要
- 事象: 公開ポータルの申込取消が失敗する。
- 影響範囲: 非会員（外部申込者）の取消API `cancelTrainingExternal`。
- 検出手段: Playwright E2E（P1-06 実行時）。

## 2. 発生条件と再現手順
1. `@75` が反映された公開ポータル（`.../exec?app=public`）で、外部申込を1件作成する。
2. 作成した申込IDと一致メールアドレスで取消を実行する。
3. 取消処理が失敗し、以下のエラーを表示する。  
   `セル E8 に入力したデータは、このセルのデータ入力規則に違反しています。`

## 3. 原因（Root Cause）
- `backend/Code.gs` の `cancelTrainingExternal_()` で、`T_研修申込.申込状態コード` に `取消` を書き込んでいた。
- ただし当該列の正規値はコード値（`APPLIED` / `CANCELED`）であり、表示文言の `取消` は入力規則違反。
- 結果としてスプレッドシート書き込み時に例外となり、取消API全体が失敗していた。

## 4. 解決内容
- 修正箇所: `backend/Code.gs`（`cancelTrainingExternal_`）
- 変更内容: `申込状態コード: '取消'` を `申込状態コード: 'CANCELED'` に修正。
- 反映:
  - `npm run build:gas`
  - `npx clasp push --force`
  - `npx clasp version "fix: external cancel status code CANCELED"` で `@76` 作成
  - 固定2 Deployment ID を `@76` に更新

## 5. 検証結果
- `npx clasp deployments` で会員/公開の固定2IDがともに `@76` を指すことを確認。
- 同一手順で再試行し、公開取消が成功することを確認（P1-06 PASS）。

## 6. 再発防止
- ステータス列には表示文言ではなくコード値を保存する方針を明文化する。
- 取消系テスト（公開/会員）をデプロイ前の必須回帰に含める。
- 可能であれば、書き込み前にコード値のホワイトリスト検証を追加する。

## 6.1 追加整合修正（2026-03-14）
- 取消側だけでなく、公開申込側の新規書込でもコード値統一を実施。
- 修正内容:
  - `applyTrainingExternal_()` の `申込状態コード` を `申込済` から `APPLIED` へ変更。
- 目的:
  - `M_申込状態`（`APPLIED` / `CANCELED`）との完全整合を維持し、表示文言混在を排除する。
- 反映:
  - `@77` を作成し、固定2 Deployment ID を同時に `@77` へ更新。
