# 開発ルールブック（絶対基準）

更新日: 2026-03-10  
対象: 枚方市介護支援専門員連絡協議会 会員システム

## 1. 目的
- 開発・デプロイ・運用判断のブレをなくす。
- 担当者が変わっても同じ判断ができる状態を維持する。

## 2. 文書の優先順位（上位優先）
1. 本書（`docs/12_ENGINEERING_RULEBOOK.md`）
2. `docs/10_SOW.md`
3. `docs/09_DEPLOYMENT_POLICY.md`
4. `docs/05_AUTH_AND_ROLE_SPEC.md`
5. その他の設計/運用文書

矛盾時は上位文書を正とし、下位文書を同日中に修正する。

## 3. 絶対ルール（Must）
- 本番URL固定運用を維持する（固定Deployment ID運用）。
- デプロイ前に `docs/09_DEPLOYMENT_POLICY.md` と `docs/10_SOW.md` を再確認する。
- DB不整合を禁止する（型定義・マスタ・シート列・API実装を同時整合）。
- 認証仕様を厳守する。
  - 管理者ログイン: Googleアカウント
  - 会員ログイン: ID/パスワード
- 仕様変更時は、実装と同時に正本ドキュメントを更新する。
- 本番反映後は `/exec` の疎通確認結果を記録する。

## 4. 禁止ルール（Must Not）
- 事前合意なしに本番Deployment IDを変更しない。
- 404未解消のまま本番完了としない。
- ドキュメント更新なしで仕様変更を完了扱いにしない。

## 5. 例外運用ルール
- 例外を使えるのは、標準手順で復旧不能と確認できた場合のみ。
- 例外実行前に、以下を必ず記録する。
  1. 症状
  2. 実施した標準切り分け
  3. 復旧不能の根拠
  4. 例外内容
  5. 承認者
- 例外実施後は、本書と関連手順書の固定値/運用値を同時更新する。

## 6. 本番デプロイ・ゲート（完了条件）
1. `npm run build:gas` 成功
2. `npx clasp push --force` 成功
3. 新Version作成成功
4. 固定Deployment IDへのVersion適用（標準）
5. `/exec` 疎通確認（404でない）
6. GitHubへのコミット/プッシュ
7. 変更ドキュメント更新

上記 1〜7 が揃って初めて「本番完了」とする。

## 7. 変更管理（推奨）
- 重要変更は `docs/` に「決定記録（Decision Record）」を追加する。
- 仕様変更は、以下を1セットで管理する。
  - 仕様（何を変えるか）
  - 実装（どこを変えたか）
  - 検証（どう確認したか）
  - 反映（どのVersion/Deploymentか）
