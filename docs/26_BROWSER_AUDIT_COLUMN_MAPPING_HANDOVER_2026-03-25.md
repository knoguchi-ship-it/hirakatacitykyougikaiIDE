## 1. 結論

- **現時点の live DB に、ホワイトリスト以外で「ヘッダに対して全行が1列ずつ右左にズレている」タイプの物理的カラムズレは確認できなかった。**

- ユーザーが「カラムズレ」に見ている違和感の主因は、**移行マッピングの設計問題**である可能性が高い。
- ただし、**ホワイトリストだけは過去に物理的列ズレ事故が実在**し、その原因は `writeSheetHeaders_()` がヘッダだけ上書きしデータ行を移動しなかったことだった。これは個別補修関数 `repairWhitelistData()` が存在する。

## 3. 原因解析

### 3.1 物理的な列ズレではなく、値マッピングの問題

`T_会員` の値をブラウザ経由で source / target 比較した結果、典型例として以下が確認できた。

#### 例: 押江 朋子

source:

- `郵送先 = 自宅`
- `メールアドレス = 08083121267@docomo.ne.jp`
- `勤務先 = (有)悠久 悠久訪問看護ステーション`
- `CM番号 = 13120243`
- `郵便番号 = 576-0051`
- `連絡先住所 = 交野市倉治8丁目58-14`
- `連絡先電話番号 = 080-8312-1267`
- `その他連絡先 = 空`

target (`T_会員.会員ID=91788667`):

- `代表メールアドレス = 08083121267@docomo.ne.jp`
- `勤務先名 = (有)悠久 悠久訪問看護ステーション`
- `勤務先電話番号 = 080-8312-1267`
- `携帯電話番号 = 空`
- `自宅郵便番号 = 576-0051`
- `自宅都道府県 = 大阪府`
- `自宅市区町村 = 交野市`

これは「1列ずれた」のではなく、移行コードが以下の規則で値を置いているために起きている。

- 個人会員の `携帯電話番号` は **常に** source `その他連絡先(U列)` から入れる
- 個人会員の `勤務先電話番号` は **常に** source `連絡先電話番号(S列)` から入れる
- 住所は `郵送先` を見て `自宅*` か `勤務先*` に振り分ける

該当コード:

- 個人会員生成: `backend/Code.gs` 8921-8978 付近
- `携帯電話番号: r.rawOtherContact`
- `勤務先電話番号: r.rawPhone`
- `自宅/勤務先住所` の振り分け: `r.rawMailDest`

### 3.2 見た目違和感を生む設計

現行移行では、個人会員でも `勤務先名` は保持しつつ、住所だけ `郵送先=自宅` なら `自宅*` に入る。
そのため、次のような見え方が発生する。

- 勤務先名は入っている
- 勤務先電話番号にも値が入っている
- 住所は自宅側に入っている

これはユーザー目線では「カラムがズレた」ように見えるが、実際は **一つの source 列を複数意味で解釈している** ことが原因。

### 3.3 事業所会員の代表者名は設計通り

事業所会員の `T_会員` 行には、代表者スナップショットが入る。

例:

- `T_会員.会員ID=4539021` は `友田 善隆`
- `T_事業所職員.職員ID=09740b4f-c1a5-4248-a0d6-8ffabf08810c` は `野口 健太`

これは列ズレではなく、`T_会員 = 事業所本体 + 代表者表示情報`、`T_事業所職員 = 個別職員` という現スキーマの結果。

### 3.4 既知の本物の列ズレ事故はホワイトリストだけ

`backend/Code.gs` のコメント通り、`repairWhitelistData()` は以下の事故用補修関数。

- `writeSheetHeaders_()` がヘッダだけ上書き
- データ行を新列順へ移していなかった
- 結果として `T_管理者Googleホワイトリスト` に物理的列ズレが発生

該当コード:

- `backend/Code.gs` 375-410 付近
- `backend/Code.gs` 7146-7163 付近

一方、現在の `normalizeTableColumns_()` は旧ヘッダ名を見て列単位で再配置しており、同事故を抑止する実装になっている。

## 4. 今回の監査で追加確認したリスク

### 4.1 source のメールアドレスは一意ではない

source `★会員名簿 / 2025年度` にはメール重複がある。

- 重複メール件数: 9
- 例:
  - `watashinoiesougoujigyou@yahoo.co.jp` x4
  - `kousei-3@outlook.jp` x2
  - `higashicare@hanamaru-day.com` x2

したがって、**source と target の突合・補修をメールアドレスだけで実施してはいけない。**
今後の監査・補修・再移行では、最低でも以下の複合キーが必要。

- `会員種別`
- `氏名`
- `勤務先`
- `CM番号`
- `ソース行番号`

## 5. 解決案

### 案A: 既存スキーマを維持し、移行マッピングだけ修正して再移行

最も現実的。

修正候補:

1. 個人会員の電話マッピングを見直す
   - 現行: `S=連絡先電話番号 -> 勤務先電話番号`, `U=その他連絡先 -> 携帯電話番号`
   - 提案: `郵送先=自宅` かつ `Uが空` かつ `Sが携帯番号形式(070/080/090)` の場合は `携帯電話番号` に入れる
2. 個人会員の `勤務先電話番号` は、`勤務先名` が空でない場合のみ原則採用する
3. `発送方法コード` はメール有無から自動推定せず、source `発送(H列)` の意味を整理して明示変換する
4. `_MIGRATION_MAP` だけでなく、`_MIGRATION_DIFF` 的な監査シートを追加し、source -> target の主要列変換結果を残す

### 案B: スキーマを見直す

中長期案。

1. `T_会員` に `主連絡先電話番号` を追加する
2. `勤務先電話番号` / `携帯電話番号` / `自宅電話番号` の意味を厳密化する
3. `T_事業所職員` にも `姓`, `名`, `セイ`, `メイ` を追加する

ただし本番影響が大きいため、今すぐなら案Aが先。
