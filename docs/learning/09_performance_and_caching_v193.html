GAS V8 ランタイム

GAS が使っている JavaScript 実行エンジンの最新情報と、なぜ重要かを理解します。

V8 とは何か

V8 は Google Chrome・Node.js・Deno が使っている JavaScript 実行エンジン（JIT コンパイラ）です。 GAS は 2020 年に旧エンジン「Rhino」から V8 に移行しました。

比較旧 Rhino エンジン現行 V8 エンジン
JavaScript バージョン ES5 相当（古い） ES2020+ 対応（モダン）
速度（純粋 JS 処理） 遅い 最大 10 倍高速
速度（Workspace API 呼び出し） やや速い わずかに遅い
async/await 使えない ✅ 使える
サポート終了 2026年1月31日 廃止 現行標準
⚠️
2026年1月31日: Rhino 廃止確定
Google は 2025年2月20日に Rhino を非推奨とし、2026年1月31日以降は Rhino スクリプトの実行が完全停止されることを発表済みです。本システムはすでに V8 移行済みのため問題ありません。
💡
V8 で Workspace API がわずかに遅い理由
V8 の JIT コンパイルによるオーバーヘッドが Spreadsheet API などの呼び出し前処理に影響するためです。ただし、これは「バッチ処理」「キャッシュ」で十分に補えます。

コールドスタート

GAS が遅い最大の原因。コンテナの仕組みを理解することが高速化の第一歩です。

なぜコールドスタートが発生するのか

GAS の実行環境（コンテナ）は、一定時間（目安: 数分〜十数分）使用されないと自動的に破棄されます。 次のリクエストが来ると、GAS は新しいコンテナを立ち上げる必要があり、 この「初回起動コスト」がコールドスタートです。

コールドスタート vs ウォームスタートの比較

❄️ コールドスタート（0 → 約20〜26秒）

  1. V8 コンテナ起動
    仮想マシン初期化 〜3s
  2. Code.gs コンパイル
    9,000行+をパース・JIT 〜5〜10s
  3. 関数実行
    Spreadsheet 読み込み等 〜3〜5s

🔥 ウォームスタート（〜1〜2秒）

  1. 既存コンテナ再利用
    起動コスト: 0
  2. 関数実行
    CacheService キャッシュヒット

ウォームアップ戦略（v188 実装済み）

GAS の時間ベーストリガーで warmUp() を 5〜10 分おきに実行します。 誰もアクセスしていない時間帯でもコンテナを生き続けさせ、次のアクセス時のコールドスタートを防ぎます。

function warmUp() {
  // Spreadsheet に接続し、全データをキャッシュに格納

ℹ️
トリガー設定の場所
Apps Script の管理画面 →「トリガー」→「トリガーを追加」→ 関数: warmUp、イベントの種類:「時間ベースのタイマー」→「5 分おきに実行」
💡
ポイント: コールドスタート自体（〜10秒）はどうしても避けられません。しかし「頻繁にウォームアップする＋キャッシュを使う」ことでウォーム時を高速化でき、使用頻度の高い業務時間帯はほぼウォーム状態を維持できます。

大きなデータを小さなブロック（チャンク）に分割して保存・送受信する手法。CacheService の 100 KB 制限を超えるデータを複数のキーに分けて保存し、読み出し時に結合します。本システムでは putChunkedCache_ / getChunkedCache_ ヘルパー関数で実装しています。

Spreadsheet API とバッチ読み込み

スプレッドシートへのアクセスは「GAS → Google サーバー → スプレッドシート」というネットワーク往復が伴います。呼び出し回数を最小化することが最も効果的な最適化です。
❌ 遅いパターン（セル単位・1,000 回の API）
100行 × 10列 = 1,000回の往復
for (var i = 1; i <= 100; i++) {
  var val = sheet
    .getRange(i, 1)
    .getValue(); // 毎回ネットワーク往復
}
✅ 速いパターン（一括取得・1 回の API）
全データを 1 回で取得
var values = sheet
  .getRange(1, 1,
    sheet.getLastRow(),
    sheet.getLastColumn())
  .getValues(); // 1回だけ
values.forEach(function(row) {
  // JS 上で処理（超高速）
});

Google の公式ベンチマーク（2025年データ）では、getValues() の一括取得は getValue() のセル単位取得と比べて 20倍以上高速になります。5,000 セルを 1 回で取得すれば 0.5 秒以下ですが、100 回の個別取得では 10 秒以上かかることもあります。

💡
原則: スプレッドシートへの「読み書きは 1 回にまとめる」。これが GAS パフォーマンス最適化の最重要ルールです。

CacheService の仕組み

CacheService は「一度読んだデータを一時保存して次回は読み直しを省く」ためのインメモリキャッシュです。業界標準の Cache-Aside パターンを実装します。
インメモリキャッシュ（In-Memory Cache）
RAM（メモリ）上に保存したデータのこと。HDD/SSD からの読み込みの 1,000〜10,000 倍、ネットワーク越しの読み込みの 10〜100 倍高速です。Redis・Memcached などが業界標準。GAS の CacheService はこれの簡易版です。

3 種類のキャッシュスコープ

メソッド共有範囲主な用途本システム
getScriptCache() 全ユーザー共通 マスターデータ・設定値 ✅ 会員データ・管理者ホワイトリスト
getUserCache() ログイン個人のみ ユーザー固有のセッション情報 未使用
getDocumentCache() 特定ドキュメント内 Sheets/Docs に紐付くデータ 未使用

Cache-Aside パターン（基本実装）

Cache-Aside パターン（業界標準）

function getDataWithCache_() {
  var cache = CacheService.getScriptCache();
  var CACHE_KEY = 'my_data_v1';
  var TTL_SECONDS = 600; // 10 分

  // ① キャッシュを確認（null = ミス or 期限切れ）
  var cached = cache.get(CACHE_KEY);
  if (cached) {
    return JSON.parse(cached); // ✅ キャッシュヒット → 即返す
  }

  // ② キャッシュミス → Spreadsheet から読み込む（遅い）
  var data = readFromSpreadsheet_();

  // ③ 次回のためにキャッシュに保存（JSON 文字列化が必要）
  try {
    cache.put(CACHE_KEY, JSON.stringify(data), TTL_SECONDS);
  } catch(e) {
    // 100KB 超えでエラー → チャンキングが必要
  }
  return data;
}

// データ更新時はキャッシュを無効化（キャッシュ無効化 = Cache Invalidation）
function updateDataAndInvalidate_(newData) {
  saveToSpreadsheet_(newData);
  cache.remove(CACHE_KEY); // ← 古いキャッシュを削除
}
⚠️
CacheService は保証しない
公式ドキュメント（2026年3月更新）によると、「キャッシュはデータが期限切れになる前に削除される場合がある（メモリ圧迫等）」と明記されています。cache.get() が返す値は必ず null チェックしてください。

backend/Code.gs — checkAdminBySession_（v193 抜粋）

function checkAdminBySession_() {
  // ① GAS サーバー側でブラウザログイン中の Google メールを取得
  var email = Session.getActiveUser().getEmail();
  if (!email) throw new Error('Googleアカウントでログインしていません。');
  email = email.toLowerCase();

  var ss = getOrCreateDatabase_();
  var cache = CacheService.getScriptCache();

  // ② ホワイトリストをキャッシュから取得（なければ Spreadsheet から読む）
  var whitelistRows;
  var cachedWL = cache.get('admin_wl_v1');
  if (cachedWL) {
    try { whitelistRows = JSON.parse(cachedWL); } catch(e) {}
  }
  if (!whitelistRows) {
    whitelistRows = getRowsAsObjects_(ss, 'T_管理者Googleホワイトリスト');
    try { cache.put('admin_wl_v1', JSON.stringify(whitelistRows), 300); } catch(e) {}
  }

  // ③ 認証アカウントも同様にキャッシュを利用
  var authRows;
  var cachedAuth = cache.get('admin_auth_v1');
  if (cachedAuth) {
    try { authRows = JSON.parse(cachedAuth); } catch(e) {}
  }
  if (!authRows) {
    authRows = getRowsAsObjects_(ss, 'T_認証アカウント')
      .filter(function(r) { return !toBoolean_(r['削除フラグ']); });
    try { cache.put('admin_auth_v1', JSON.stringify(authRows), 300); } catch(e) {}
  }

  // ④ 会員名は fetchAllDataFromDb_() のキャッシュから取得（あれば）
  var cachedAllData = getChunkedCache_(cache, getAllDataCacheKey_());
  if (cachedAllData && cachedAllData.members) {
    for (var i = 0; i < cachedAllData.members.length; i++) {
      if (cachedAllData.members[i].id === memberId) {
        memberName = cachedAllData.members[i].lastName + ' ' + cachedAllData.members[i].firstName;
        break;
      }
    }
  }
  // キャッシュにない場合のみ Spreadsheet を直接読む（フォールバック）
}

deflate-raw 圧縮の仕組み

なぜ JavaScript コードが 60% 以上小さくなるのか、圧縮の原理から理解します。

DEFLATE アルゴリズム

DEFLATE は 1993 年に策定された業界標準の可逆圧縮アルゴリズムです。 ZIP・PNG・HTTP gzip 転送など、日常的に使われているほぼすべての圧縮フォーマットの核心です。 2 つの技術を組み合わせています：

  • LZ77（辞書圧縮）: 同じパターンが繰り返し現れる部分を「位置と長さの参照」に置き換える
  • Huffman 符号: 出現頻度の高い文字を短いビット列で表現する
💡
JavaScript が特に圧縮効率が高い理由
JavaScript コードには function・var・return・const などのキーワードや、React の場合は同じコンポーネントの繰り返しパターンが多く含まれます。LZ77 が「同じ文字列の繰り返し」を得意とするため、JavaScript は特に高い圧縮効率（60〜80%）が得られます。

3 種類の圧縮フォーマットの違い

フォーマットヘッダーチェックサム主な用途ブラウザ API
gzip あり（gzip 形式） CRC-32 HTTP gzip 転送 DecompressionStream('gzip')
deflate あり（zlib 形式） Adler-32 HTTP Content-Encoding DecompressionStream('deflate')
deflate-raw なし なし 埋め込みデータ・カスタム圧縮 DecompressionStream('deflate-raw')

本システムで deflate-raw を使う理由: ヘッダー・チェックサムがない分だけペイロードを最小化でき、ブラウザの DecompressionStream API でネイティブに展開できるからです。

Base64 エンコードが必要な理由

Base64 エンコード
圧縮後のデータはバイナリ（0〜255 の数値の羅列）です。HTML や JavaScript の文字列の中にバイナリをそのまま書くと文字化けやパースエラーが起きます。Base64 は「3 バイトを 4 文字の英数字に変換」することでバイナリを安全なテキストに変換します。変換後サイズは約 33% 増加しますが、HTML に埋め込めるようになります。

Web Streams API と DecompressionStream

ブラウザ側で圧縮データを展開する仕組みです。2023 年に全ブラウザ対応が完了した新しい Web 標準 API です。

Web Streams API の 3 つのコア

📥
ReadableStream
読み取り元。Fetch レスポンス・ファイル・バイト配列などをストリームとして扱う。
⚙️
TransformStream
変換処理。入力ストリームを変換して出力するパイプ。DecompressionStream はこれを実装している。
📤
WritableStream
書き込み先。データの最終的な出力先（ファイル・DOM など）を定義する。

ブラウザ側の展開フロー（実装コード）

ブラウザ実行時デコンプレッサー（HTML に埋め込まれる）

(async function(){
  // ① <script id="__app_data__"> から Base64 テキストを取得
  var d = document.getElementById('__app_data__').textContent.trim();

  // ② Base64 デコード → バイナリ文字列 → Uint8Array（バイト配列）
  var b = atob(d), n = new Uint8Array(b.length);
  for (var i = 0; i < b.length; i++) n[i] = b.charCodeAt(i);

  // ③ Uint8Array → ReadableStream（Fetch API の Response 経由）
  // ④ .pipeThrough() で DecompressionStream（deflate-raw 展開）を適用
  // ⑤ 展開後ストリームを Response.text() で文字列化
  var t = await new Response(
    new Response(n).body
      .pipeThrough(new DecompressionStream('deflate-raw'))
  ).text();

  // ⑥ 展開された JavaScript コードを実行 → React アプリ起動
  new Function(t)();
})();
pipeThrough（パイプスルー）
ReadableStream から流れてくるデータを TransformStream に通す操作。Unix シェルの |（パイプ）と同じ概念です。stream.pipeThrough(transformer) でデータを変換しながら流すことができます。

GAS の CSP 制約と unsafe-eval

v189 で「画面真っ白」が起きた原因がここにあります。GAS のセキュリティ制約を正しく理解することが重要です。

CSP（Content Security Policy）とは

Content Security Policy（CSP）
Web ページが「どのリソース・スクリプトの実行を許可するか」をブラウザに宣言するセキュリティ仕様。HTTP レスポンスヘッダーで設定します。XSS（クロスサイトスクリプティング）攻撃を防ぐために設計されています。

例: Content-Security-Policy: script-src 'self' 'unsafe-eval'

GAS の iframe が使う CSP

GAS の HTML Service は script.googleusercontent.com の iframe 内で動作します。この iframe には Google によって自動的に CSP が付与されます。

CSP ディレクティブGAS での状態意味
'unsafe-eval' ✅ 許可 eval()・new Function() などの文字列→コード変換を許可
'unsafe-inline' ✅ 許可 HTML 内 <script> タグの直書きスクリプトを許可
blob: URI ❌ ブロック URL.createObjectURL(blob) で生成した URL からのスクリプト読み込みを禁止
data: URI ❌ ブロック data:text/javascript,... 形式の URL を禁止

v189 の失敗と v191 の解決

❌ v189 — blob: URI → GAS CSP でブロック
vite-plugin-singlefile-compression の動作
// 圧縮済み JS を Blob URL に変換して動的インポート
const blob = new Blob([jsCode], {
  type: 'text/javascript'
});
const url = URL.createObjectURL(blob);
import(url); // ← GAS CSP が blob: をブロック
// 結果: 画面真っ白（コンソールエラー）
✅ v191 — new Function() → unsafe-eval で許可
カスタム compress-html.mjs の動作
// 展開した JS 文字列を直接コードとして実行
var t = await /* deflate-raw で展開した JS */;
new Function(t)();
// ← unsafe-eval が許可 → 正常動作
// 結果: 通常の起動 + 60% 圧縮を両立
🔒
new Function() はここでは安全
一般的に new Function() は「外部からの入力をコードとして実行する」場合に危険です。しかし本システムでは「npm run build:gas で自分が生成した、検証済みの JavaScript コード」のみを実行しています。外部ユーザーの入力は一切実行していないため、セキュリティリスクはありません。

なぜ gzip 転送と同等の効果か

💡
GAS はサーバー側 gzip を行わない
通常の Web サーバー（Nginx・Apache・CDN）は HTTP レスポンスを gzip 圧縮して転送し、ブラウザが自動展開します。GAS にはこの機能がないため、「ビルド時に圧縮済みの HTML を送る」ことが通常の gzip 転送と同等の効果を生みます。raw HTML サイズの削減 = 実際のダウンロード量の削減に直結します。

管理者認証フロー

Google Identity Services（GIS）が使えない理由

通常の Web アプリでは「Google でサインイン」ボタンを GIS（Google Identity Services）で実装できます。 しかし GAS の iframe は script.googleusercontent.com というオリジン上で動作するため、 OAuth 2.0 の認証コールバック URL として登録することができません。

オリジン（Origin）
プロトコル://ホスト名:ポート番号 の組み合わせ（例: https://script.googleusercontent.com）。ブラウザのセキュリティポリシーは「同一オリジン」を基準に動作します。OAuth は正確なコールバック URL の登録が必要ですが、GAS の iframe オリジンは登録不可能です。

Session.getActiveUser() による解決

google.script.run 経由で GAS 関数を呼び出すと、 GAS サーバー側で呼び出したブラウザにログイン中の Google アカウントを取得できます。 これを Session.getActiveUser().getEmail() で取得します。

⚠️
空文字が返されるケース（公式: 2025年12月更新）
  • スクリプト実行設定が「Anyone（全ユーザーとして実行）」の場合
  • 呼び出しユーザーが組織外のアカウントで、セキュリティポリシーが制限している場合
  • doGet() など、ユーザー認証なしに実行されるコンテキスト内
本システムは appsscript.json で execute as: USER_DEPLOYING・access: DOMAIN に設定しているため正常動作します。
管理者ログインフロー（v193）
  1. ボタン押下 → google.script.run
    ブラウザが checkAdminBySession を GAS サーバーに送信
  2. Session.getActiveUser().getEmail()
    GAS がブラウザのログイン中 Google アカウントを取得（クライアント送信値を信頼しない）
  3. CacheService 確認
    admin_wl_v1 キャッシュヒット → Spreadsheet 読み込みをスキップ
  4. ホワイトリスト照合
    メールアドレスが T_管理者Googleホワイトリストに存在するか確認。なければエラー（ログにも記録）
  5. 認証成功 → 権限コード返却
    MASTER / ADMIN / TRAINING_MANAGER 等の権限コードと displayName をブラウザに返す
🔒
セキュリティ上の重要ポイント
メールアドレスはブラウザからの入力ではなく、GAS が Google SSO から直接取得します。クライアント側でメールアドレスを偽造してもサーバー側では Session.getActiveUser() の値だけを使うため、なりすましは不可能です。

認証とデータ取得の分離（v192）

「ログインしたら全データを一気に取ってくる」設計が、管理者ログインを 26 秒にした原因でした。
❌ v192 以前: 認証 + データ取得を同時実行
App.tsx（旧）
// 1 回の API で認証 + 全会員データ取得
const result = await api.adminLoginWithData();
// ↑ checkAdminBySession_() +
//   getMemberPortalData_() を同時呼び出し
// → 両方で 15〜18 秒かかる
✅ v192 以降: 認証のみ → データは遅延取得
App.tsx（新）
// 認証のみ（高速）
const auth = await api.checkAdminBySession();
// 会員データは各ページ遷移時に必要分だけ取得
// → ログイン完了が大幅高速化
遅延取得（Lazy Loading）
必要になったときにだけデータを取得する設計手法。「最初に全部読む」よりもログイン体験が速くなります。Netflix の動画リストを全部最初に読み込まず、スクロールするたびに追加取得するのと同じ概念です。

残課題: B-02 — fetchAllDataFromDbFresh_ 最適化

キャッシュが切れた（コールドスタート後・10 分経過後）の初回データ読み込みをさらに高速化する Phase 3 の課題です。

現状の問題

fetchAllDataFromDbFresh_() は 6 シートを順次読み込みます。 各シートで getLastRow() + getLastColumn() + getValues() = 3 回の API 呼び出しが発生するため、合計 18 回の Spreadsheet API 呼び出しになります。

❌ 現状: 18 回の API 呼び出し
各 getRowsAsObjects_ で 3 回ずつ
function fetchAllDataFromDbFresh_() {
  var ss = getOrCreateDatabase_();
  var memberRows   = getRowsAsObjects_(ss, 'T_会員');       // 3回
  var staffRows    = getRowsAsObjects_(ss, 'T_事業所職員');   // 3回
  var authRows     = getRowsAsObjects_(ss, 'T_認証アカウント');// 3回
  var trainingRows = getRowsAsObjects_(ss, 'T_研修');        // 3回
  var appRows      = getTrainingApplicationRows_(ss, ...);   // 3回
  var feeRows      = getRowsAsObjects_(ss, 'T_年会費納入履歴');// 3回
  // 合計: 18 回の API 呼び出し
}
✅ 改善案: ss.getSheets() で全シート一括取得
シート参照を使い回す
function fetchAllDataFromDbFresh_Opt() {
  var ss = getOrCreateDatabase_();

  // ✅ 全シートを一括取得（1 回の API 呼び出し）
  var sheets = {};
  ss.getSheets().forEach(function(s) {
    sheets[s.getName()] = s;
  });

  // getSheetByName の代わりに直接参照 → 追加の API 呼び出し不要
  var memberRows = getRowsFromSheet_(sheets['T_会員']);
  // ... 推定 18 回 → 7〜8 回に削減
}
