3. V8 ランタイムとコールドスタート

V8 ランタイムとは？

V8 は Google Chrome ブラウザと Node.js が使っている JavaScript 実行エンジンです。 GAS は 2020 年に従来の「Rhino」エンジンから V8 へ移行しました。 これにより Arrow Function・async/await・Spread Operator など モダンな JavaScript が使えるようになりました。

重要: 2026年1月31日に Rhino は廃止
Google は 2025年2月20日に Rhino ランタイムを非推奨とし、 2026年1月31日以降は Rhino スクリプトの実行が停止されることを発表しました。 このシステムはすでに V8 移行済みのため問題ありません。

コールドスタートの仕組み

GAS の実行環境（コンテナ）は、一定時間（目安: 数分〜数十分）使用されないと破棄されます。 次のリクエストが来たとき、GAS は新しいコンテナを立ち上げ、 V8 エンジンを初期化してスクリプトを読み込み直す必要があります。 この「再起動」の時間を コールドスタート と呼びます。

GAS の「時間ベーストリガー」を使って、warmUp() 関数を 5〜10 分おきに自動実行します。 これにより、誰もアクセスしていない時間帯でもコンテナを生かし続け、 次の実際のアクセス時にコールドスタートを避けます。

4. GAS の制約・クォータ

制約項目 値 本システムへの影響
スクリプト最大実行時間 6 分 管理者コンソールの全データ取得が長時間かかると超過リスク
CacheService 1エントリ上限 100 KB 全データ（176会員）は 100 KB 超のため「チャンキング」で分割保存
CacheService キャッシュ最大数 1,000 エントリ チャンクを複数エントリに分けて保存する方式で対応
CacheService 最大 TTL 6 時間（21,600 秒） 本システムは 600 秒（10分）を設定
Spreadsheet 1回の読み込みコスト 数百 ms〜1 秒 6 シートを順次読むと 3〜6 秒かかる
GAS フロントエンド配信 CDN なし・毎回動的生成 HTML 圧縮で転送量を削減することが有効
チャンキング（Chunking）とは
大きなデータを小さなブロック（チャンク）に分けて保存・送受信する手法。 CacheService の 100 KB 制限を超えるデータは、複数のキーに分割して保存し、 読み取り時に結合して元のデータに戻します。 本システムでは putChunkedCache_ / getChunkedCache_ 関数で実装しています。

5. Spreadsheet API とバッチ処理

なぜ Spreadsheet 読み込みが遅いのか

スプレッドシートへのアクセスは、GAS から見るとリモートの Web API を呼び出すのと同等です。 1回の操作につき数百 ms のラウンドトリップが発生します。

❌ 遅いパターン（セル単位）
// 100行 × 10列 = 1,000回の API 呼び出し
for (var i = 1; i <= 100; i++) {
  var val = sheet.getRange(i, 1).getValue();
  // ↑ 毎回 API 呼び出し = 超遅い
}
✅ 速いパターン（一括取得）
// 1回の API 呼び出しで全取得
var values = sheet
  .getRange(1, 1, sheet.getLastRow(), sheet.getLastColumn())
  .getValues(); // ← 1回だけ
// JavaScript 上で処理
values.forEach(function(row) { /* ... */ });

Google の公式ベンチマークでは、getValues() の一括取得は getValue() の単セル取得と比べて 20倍以上高速になる場合があります。

本システムの実装パターン

// backend/Code.gs — getRowsAsObjects_ (ヘルパー関数)
// シート全体を1回で読み込み、オブジェクト配列に変換する
function getRowsAsObjects_(ss, sheetName) {
  var sheet = ss.getSheetByName(sheetName);
  if (!sheet) return [];

  var lastRow = sheet.getLastRow();
  var lastCol = sheet.getLastColumn();
  if (lastRow < 2) return [];

  // ✅ 全データを1回で取得（A1:最終列 の範囲）
  var values = sheet
    .getRange(1, 1, lastRow, lastCol)
    .getValues();

  var headers = values[0]; // 1行目がヘッダー
  var rows = [];
  for (var i = 1; i < values.length; i++) {
    var obj = {};
    for (var j = 0; j < headers.length; j++) {
      obj[headers[j]] = values[i][j];
    }
    rows.push(obj);
  }
  return rows;
}

6. CacheService の仕組み

CacheService とは

GAS が提供する組み込みのインメモリキャッシュ機能です。 スプレッドシート等のデータを一時的に保存し、次の呼び出し時に高速に取得できます。

インメモリキャッシュ（In-Memory Cache）とは
「メモリ上（RAM）に保存されたデータ」のこと。ディスク（HDD/SSD）やネットワーク越しの読み込みより 圧倒的に高速（10〜1,000倍）です。ただしサーバー再起動や一定時間経過で消えます。 Redis・Memcached などが業界標準のインメモリキャッシュシステムです。

3 種類のキャッシュスコープ

メソッド スコープ 用途 本システムでの使用
getScriptCache() スクリプト全体で共有
（全ユーザー共通）		 全ユーザー共通のマスターデータ ✅ 会員データ全体・管理者ホワイトリスト
getUserCache() ログインユーザー個人 ユーザー固有のセッション情報 現在未使用
getDocumentCache() 特定ドキュメント内 Sheets・Docs に紐付いたデータ 現在未使用

基本的な使い方

var cache = CacheService.getScriptCache();

// ① 保存（put）
// キー, 値（文字列のみ）, TTL（秒）
cache.put('my_key', JSON.stringify(data), 300); // 5分間保持

// ② 取得（get）— null が返ればキャッシュ切れ
var cached = cache.get('my_key');
if (cached) {
  return JSON.parse(cached); // キャッシュヒット → 高速
} else {
  // キャッシュミス → Spreadsheet から読み込み
  var data = readFromSpreadsheet_();
  cache.put('my_key', JSON.stringify(data), 300);
  return data;
}

// ③ 削除（remove）— データが更新されたとき
cache.remove('my_key');

キャッシュの制約と注意点

重要な制限
  • 1エントリ最大 100 KB — 大きなデータは「チャンキング」で分割保存が必要
  • 最大 1,000 エントリ — キャッシュ満杯時は古いエントリ約 10% が自動削除
  • 文字列のみ保存可 — オブジェクトは JSON.stringify() で変換が必要
  • TTL は最大 6 時間（21,600 秒）
  • 保証されない — GAS はキャッシュをいつでも強制削除する権利を持つ（メモリ圧迫時など）
キャッシュ戦略のパターン: Cache-Aside（キャッシュアサイド）
本システムで使っているパターン。「まずキャッシュを確認 → ミスなら直接取得して保存」という流れ。 Redis などでも標準的に使われる業界定番のキャッシュ戦略です。

function checkAdminBySession_() {

  if (!email) throw new Error('Googleアカウントでログインされていません。');

  var whitelistRows;
  var cachedWL = cache.get('admin_wl_v1');

  // ④ 認証アカウントも同様にキャッシュを活用

  // ⑤ 会員名は fetchAllDataFromDb_() のキャッシュから取得（既に存在する場合）
  var cachedAllData = getChunkedCache_(cache, getAllDataCacheKey_());
  if (cachedAllData && cachedAllData.members) {

    for (var i = 0; i < cachedAllData.members.length; i++) {
      if (cachedAllData.members[i].id === memberId) {

        break;

  // キャッシュがない場合だけ Spreadsheet を直接読む（フォールバック）
}

キャッシュ無効化（Cache Invalidation）

キャッシュ無効化とは、データが更新されたときに古いキャッシュを削除し、 次回アクセス時に最新データを読み直させる仕組みです。 「キャッシュの整合性」を保つために不可欠です。 

// saveAdminPermission_: 管理者ホワイトリストを変更した後にキャッシュを削除
function saveAdminPermission_(payload) {
  // ... ホワイトリストの保存処理 ...

  // ✅ データが変わったのでキャッシュを即時無効化
  try { CacheService.getScriptCache().remove('admin_wl_v1'); } catch(e) {}
  return { saved: true };
}

8. deflate-raw 圧縮とは

データ圧縮の基本

データ圧縮とは、データのパターンや繰り返しを利用してファイルサイズを小さくする技術です。 JavaScript コードは文字列・キーワードの繰り返しが多いため、圧縮効率が高く、 60〜80% 削減できることがあります。

DEFLATE アルゴリズム
ZIP・gzip・PNG などで使われている業界標準の圧縮アルゴリズム（1993年策定）。 LZ77（繰り返しパターンを辞書化）と Huffman 符号（出現頻度の高い文字を短いビット列で表現）を組み合わせています。 現在も最も広く使われている可逆圧縮アルゴリズムの一つです。

圧縮フォーマットの種類

フォーマット ヘッダー チェックサム 用途 ブラウザ API
deflate あり（zlib） Adler-32 HTTP コンテンツ圧縮 DecompressionStream('deflate')
gzip あり（gzip） CRC-32 HTTP gzip 転送 DecompressionStream('gzip')
deflate-raw なし なし 埋め込みデータ・カスタム用途 DecompressionStream('deflate-raw')

本システムでは deflate-raw を使用しています。 ヘッダーもチェックサムもないため、ペイロードを最小化できます。 ブラウザの DecompressionStream API が natively サポートしているため、 展開処理を追加ライブラリなしで実現できます。

Base64 エンコードとは

圧縮後のバイナリデータ（0〜255 の数値の列）は、HTML や JavaScript 文字列の中に直接埋め込めません。 Base64 は 3 バイトを 4 文字の ASCII に変換するエンコード方式で、 バイナリを安全にテキストとして扱えるようにします。

// scripts/compress-html.mjs — ビルド時圧縮スクリプト（Node.js）
import fs from 'fs';
import zlib from 'zlib';

function compressHtmlFile(inputPath) {
  const html = fs.readFileSync(inputPath, 'utf8');

  // vite-plugin-singlefile が生成した <script type="module" crossorigin> を抽出
  const scriptRegex = /<script type="module" crossorigin>([\s\S]*?)<\/script>/;
  const match = html.match(scriptRegex);
  if (!match) return; // 対象なし

  const scriptContent = match[1];
  const scriptBytes = Buffer.from(scriptContent, 'utf8');

  // ① deflate-raw で最高圧縮（Z_BEST_COMPRESSION = レベル9）
  const compressed = zlib.deflateRawSync(scriptBytes, {
    level: zlib.constants.Z_BEST_COMPRESSION
  });

  // ② Base64 エンコード（バイナリ → ASCII テキスト）
  const encoded = compressed.toString('base64');

  // ③ HTML に埋め込む形式を生成
  const decompressor =
    `<script id="__app_data__" type="application/octet-stream">${encoded}</script>\n` +
    `<script>\n(async function(){\n` +
    `  // ブラウザ側でデコード・展開・実行\n` +
    `  var d=document.getElementById('__app_data__').textContent.trim();\n` +
    `  var b=atob(d),n=new Uint8Array(b.length);\n` +
    `  for(var i=0;i<b.length;i++)n[i]=b.charCodeAt(i);\n` +
    `  var t=await new Response(new Response(n).body.pipeThrough(new DecompressionStream('deflate-raw'))).text();\n` +
    `  new Function(t)();\n` +
    `})();\n</script>`;

  const newHtml = html.replace(scriptRegex, decompressor);
  fs.writeFileSync(inputPath, newHtml, 'utf8');
}

9. DecompressionStream API

Web Streams API とは

Web Streams API は、データを「ストリーム（連続した流れ）」として処理するブラウザ標準 API です。 大きなデータを一度にメモリに展開せず、チャンクごとに処理できます。

📥
ReadableStream
読み取り元。ファイル・ネットワーク・バイナリデータなどをストリームとして扱う。
⚙️
TransformStream
変換処理。入力ストリームを変換して出力する。DecompressionStream もこれを実装。
📤
WritableStream
書き込み先。データの最終出力先を定義する。

  // ① data 属性の script タグから Base64 テキストを取得
  var d = document.getElementById('__app_data__').textContent.trim();

  // ② Base64 → バイナリ文字列 → Uint8Array（バイト配列）に変換
  var b = atob(d), n = new Uint8Array(b.length);
  for (var i = 0; i < b.length; i++) n[i] = b.charCodeAt(i);

  // ③ Uint8Array → ReadableStream（Fetch API の Response 経由）
  // ④ pipeThrough で DecompressionStream（deflate-raw で展開）
  // ⑤ 展開後のストリームを text() で文字列化
  var t = await new Response(
    new Response(n).body.pipeThrough(new DecompressionStream('deflate-raw'))
  ).text();

  // ⑥ 展開された JavaScript コードを実行
  new Function(t)();
})();

10. GAS CSP と unsafe-eval

CSP（Content Security Policy）とは

CSP（コンテンツセキュリティポリシー）は、Web サイトが「どのリソース・スクリプトの実行を許可するか」を宣言するセキュリティ仕様です。 HTTP レスポンスヘッダーや <meta> タグで設定します。 XSS（クロスサイトスクリプティング）攻撃を防ぐために設計されています。

XSS（Cross-Site Scripting）とは
悪意のあるスクリプトを Web ページに注入して、他のユーザーのブラウザで実行させる攻撃手法。 CSP の script-src ディレクティブで実行を許可するスクリプトの出所を制限することで防ぎます。

GAS の CSP 設定と制約

GAS の HTML Service は script.googleusercontent.com 上の iframe 内で動作します。 この iframe には以下の CSP が自動的に適用されます： 

/* GAS HTML Service が自動付与する CSP（概略）*/
Content-Security-Policy:
  default-src 'self' https:;
  script-src 'self' 'unsafe-inline' 'unsafe-eval' https:;
  object-src 'none';
CSP ディレクティブ 意味 GAS での状況
'unsafe-inline' HTML 内の <script> タグや onclick="" を許可 ✅ 許可（GAS は HTML 内スクリプトを多用）
'unsafe-eval' eval()・new Function() などの文字列→コード変換を許可 ✅ 許可（React の一部実装が使用）
blob: URI URL.createObjectURL(blob) からの読み込みを許可 ❌ ブロック（v189 失敗の原因）
data: URI data:text/javascript,... からの読み込みを許可 ❌ ブロック

v189 の失敗と v191 の解決

❌ v189 の失敗アプローチ（blob: URI）

vite-plugin-singlefile-compression は展開した JS を URL.createObjectURL(blob) で Blob URL を作成し、 import(blobURL) で動的インポートしていました。 

// GAS CSP がブロック！
const blob = new Blob([jsCode], {type: 'text/javascript'});
const url = URL.createObjectURL(blob);
import(url); // ← blob: スキームはブロック

結果: 画面真っ白（Console エラー）

✅ v191 の解決アプローチ（new Function）

new Function(code)() は JS の文字列をそのままコードとして実行します。 GAS の CSP に 'unsafe-eval' が含まれているため、これは許可されます。 

// GAS CSP が許可！（unsafe-eval あり）
var t = await /* 展開した JS 文字列 */;
new Function(t)(); // ← unsafe-eval で許可

結果: 正常動作（-60% 圧縮達成）

セキュリティについての注意
new Function() や eval() は一般的に「危険な機能」として避けるべきとされています。 ただし本システムでは「自分でビルドした JavaScript コードを自分で実行する」ことが目的で、 外部からの入力をコードとして実行しているわけではありません。 GAS 側のビルドプロセス（npm run build:gas）で生成した信頼済みのコードのみを使用しているため、セキュリティリスクはありません。

なぜ圧縮しても gzip 転送量と同じくらいになるのか
通常の Web サーバーは HTTP レスポンスを gzip 圧縮して送信します（ブラウザが自動展開）。 GAS はサーバー側 gzip を行わないため、「ビルド時に圧縮済みの HTML」を送ることが 通常の gzip 転送と同等の効果を生みます。 raw HTML サイズの削減 = 実際のダウンロード量の削減に直結します。

12. 管理者認証フロー

なぜ Google ログインを使わないのか

通常の Web アプリでは Google Identity Services（GIS） を使って 「Google でログイン」ボタンを実装できます。 しかし GAS の HTML Service は script.googleusercontent.com の iframe 上で動作するため、 OAuth 2.0 の認証フローで必要な「正規のオリジン（Origin）への POST」ができません。

オリジン（Origin）とは
プロトコル://ホスト:ポート の組み合わせ。 例: https://example.com:443。ブラウザのセキュリティポリシーは「同一オリジン」を基準に動作します。 GAS は https://script.googleusercontent.com というオリジンで動作するため、 OAuth コールバック URL として登録できません。

Session.getActiveUser() を使った解決策

google.script.run 経由で GAS 関数を呼び出すと、 GAS サーバー側で呼び出したユーザーの Google アカウントのメールアドレスを取得できます。 これが Session.getActiveUser().getEmail() です。

Session.getActiveUser() の動作条件
  • Apps Script の実行設定が "Execute as: Me" であること
  • 呼び出しユーザーが 同一の Google Workspace ドメイン のアカウントであること（またはスクリプト所有者と同組織）
  • google.script.run 経由で呼び出されていること（doGet 内では空文字になる）
これらの条件が満たされないと、空文字列が返されます。
管理者ログインフロー（v193）
① ブラウザ: 「Googleアカウントで管理者ログイン」ボタン押下
google.script.run.withSuccessHandler().checkAdminBySession()

→ ブラウザにログインしている Google アカウントのメールを取得

③ キャッシュ確認: T_管理者Googleホワイトリスト をキャッシュまたは Spreadsheet から取得
メールアドレスで照合
④ ホワイトリスト照合: メールアドレスが登録済みか確認
→ 未登録なら「管理者権限がありません」エラー
⑤ 認証成功: 権限コード（MASTER/ADMIN 等）・会員ID・displayName を返却
→ App.tsx に反映、管理者コンソールを表示

v192 以前は「管理者ログイン」ボタンを押すと、 認証（checkAdminBySession_）と全会員データの取得（getMemberPortalData_）を 1回の API 呼び出しでまとめて実行していました。

13. Session.getActiveUser() の注意点

空文字が返るケース

Google の公式ドキュメント（2025年12月更新）によると、以下の場合に空文字列が返されます：

  • スクリプト実行設定が「ユーザーとして実行（Anyone）」の場合
  • 呼び出しユーザーが組織外の Google アカウントの場合（セキュリティポリシーによる）
  • 匿名アクセス（ログインしていないユーザー）
  • doGet() など、ユーザー認証なしに実行されるコンテキスト
本システムの設定
backend/appsscript.json の設定:
  • execute as: USER_DEPLOYING（スクリプト所有者として実行）
  • access: DOMAIN（同一 Google Workspace ドメインのユーザー）
この設定により Session.getActiveUser() が正しくメールを返せます。

セキュリティ設計の原則

// 管理者認証の核心部分
function checkAdminBySession_() {
  // GAS サーバー側でセッションからメールを取得（クライアントからの入力に依存しない）
  var email = Session.getActiveUser().getEmail();

  // ✅ セキュリティポイント:
  // メールアドレスはブラウザからの入力ではなく、GASが直接Google SSO から取得
  // → 偽装・なりすましが不可能

  // ホワイトリストに存在するか確認（データベースが正本）
  var matched = whitelistRows.find(function(w) {
    return w['Googleメール'].toLowerCase() === email;
  });

  if (!matched) {
    // ✅ ログインを記録して例外（攻撃の試みも記録に残る）
    appendLoginHistory_(ss, '', email, 'GOOGLE', 'FAILURE', 'ホワイトリスト未登録');
    throw new Error('管理者権限がありません。');
  }
}

15. 残課題: B-02 — fetchAllDataFromDbFresh_ 最適化

現在の課題

fetchAllDataFromDbFresh_() は、キャッシュミス時（コールドスタート直後、または 10 分経過後）に 6 つのシートを順次読み込みます。各シートの読み込みには getLastRow()・getLastColumn()・getValues() の 3 回の API 呼び出しが必要なため、合計 18 回の Spreadsheet API 呼び出しが発生します。 

// 現在の実装（6シート × 3API = 18回の API 呼び出し）
function fetchAllDataFromDbFresh_() {
  var ss = getOrCreateDatabase_();

// 改善案: ss.getSheets() で全シートを一括取得してキャッシュ
function fetchAllDataFromDbFresh_Optimized() {

  var memberRows = getRowsFromSheet_(allSheets['T_会員']);
