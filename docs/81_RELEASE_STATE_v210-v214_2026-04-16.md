## v212 — 設定保存 N+1 問題解消

### Summary

`updateSystemSettings_()` が 15 回の個別 `upsertSystemSetting_` を呼んでいた（各 2 read + 1 write）。
`getSystemSettings_()` も 12+ 回の個別 `getSystemSettingValue_` を呼んでいた。
合計 ~70 Sheets API コール（14〜35 秒）でタイムアウト相当になり、
設定保存ボタンが永遠にグレーアウトしたまま戻らない不具合を引き起こしていた。

### Changes

**GAS backend (`backend/Code.gs`)**

**新ヘルパー `getSystemSettingMap_(ss)`**
- `T_システム設定` を `getRowsAsObjects_` で 1 回だけ読み込み `{キー: 値}` マップを返す

**新ヘルパー `batchUpsertSystemSettings_(ss, updates)`**
- シートヘッダーと全データを各 1 回読み込み（合計 2 API call）
- メモリ上で全キーを更新し、`setValues()` で 1 回書き込み
- 新規キーのみ個別 `appendRow`

**リファクタリング**
- `getSystemSettings_()`: `getSystemSettingMap_` を 1 回呼ぶように変更（旧: 12+ 個別 read）
- `updateSystemSettings_()`: `batchUpsertSystemSettings_` を 1 回呼ぶように変更（旧: 15 個別 upsert）
- `getPublicPortalSettings_()`: `getSystemSettingMap_` 1 回読み込みに変更

### 改善効果
| 指標 | 旧 | 新 |
|---|---|---|
| 設定保存 API calls | ~70 | ~4 |
| 推定処理時間 | 14〜35 秒 | 1〜2 秒 |

  - trainingDefaultFieldConfigInput（全フィールド）
  - rosterTemplateSsIdInput / reminderTemplateSsIdInput / bulkMailAutoAttachFolderIdInput / emailLogViewerRoleInput
  - publicPortalTrainingMenuEnabledInput / publicPortalMembershipMenuEnabledInput
  - credentialEmailEnabledInput / credentialEmailSubjectInput / credentialEmailBodyInput

## v214 — 公開ポータル FOIC 修正・Skeleton UI 導入

### Summary

公開ポータルトップで、設定 API レスポンス到着前に全カードが一瞬表示される
**FOIC（Flash of Incorrect Content）** が発生していた。

GrowthBook `<FeaturesReady>` パターンに準拠し、設定確定前は Skeleton UI を表示し、
設定確定後に正しいカードのみ描画するよう修正。

### Root Cause

```typescript
// 旧実装: デフォルト true で全カードを即描画
const [trainingMenuEnabled, setTrainingMenuEnabled] = useState(true);
const [membershipMenuEnabled, setMembershipMenuEnabled] = useState(true);
```

API 応答到着前から `trainingMenuEnabled = true` / `membershipMenuEnabled = true` で
両カードが描画されていた。

### Changes

**Frontend (`src/public-portal/App.tsx`)**

- 初期状態を `null`（未確定）に変更:
  ```typescript
  const [trainingMenuEnabled, setTrainingMenuEnabled] = useState<boolean | null>(null);
  const [membershipMenuEnabled, setMembershipMenuEnabled] = useState<boolean | null>(null);
  ```

- `renderHome()` のカードセクションに 3 段階レンダリングを導入:
  1. `null`（設定未確定）→ Skeleton UI（`animate-pulse`、`aria-busy="true"`）
  2. 設定確定 + いずれか true → 正しいカードのみ描画
  3. 両方 false → 「🔧 現在準備中です」表示

- API 失敗時のフォールバック追加:
  ```typescript
  // 成功時: 確定値をセット
  setTrainingMenuEnabled(portalSettings.value.trainingMenuEnabled !== false);
  // 失敗時: fail-open（全表示）
  } else {
    setTrainingMenuEnabled(true);
    setMembershipMenuEnabled(true);
  }
  ```

- 研修件数表示をローディング中条件分岐から削除（Skeleton 表示中は件数不要）

### 動作フロー
```
ページ表示
  → Skeleton 2 枚（animate-pulse）表示
  → getPublicPortalSettings + getPublicTrainings 並列取得（約 1〜2 秒）
  → 設定確定 → 正しいカードのみ描画（フラッシュなし）
```

### 設計根拠（Web 調査ベース）

| 観点 | 採用パターン | 参考 |
|---|---|---|
| デフォルト | fail-conservative（null で非表示） | GrowthBook `<FeaturesReady>` |
| ローディング | Skeleton UI（animate-pulse） | Material Design 3 Skeleton |
| 失敗時 | fail-open（全表示） | 公開ポータルは可用性優先 |
| アクセシビリティ | `aria-busy="true"` + `aria-label="読み込み中"` | WCAG 2.1 Live Regions |
