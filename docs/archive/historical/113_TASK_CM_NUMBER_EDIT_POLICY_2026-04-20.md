# タスク: 介護支援専門員番号 編集ポリシーの設計と実装

作成日: 2026-04-20
ステータス: OPEN（次フェーズ着手待ち）
優先度: Medium

## 背景

v247 時点で、会員マイページにおいて事業所会員の管理者以外（一般職員・代表者を含む）が会員情報の編集・職員追加を行えるようになっている。
ただし「介護支援専門員番号（CM番号）」の修正については、以下の理由から特別な扱いが必要とユーザーが判断した。

## 問題の所在

CM番号 (`careManagerNumber`) は次の性質を持つ。

1. **業務上の一意識別子**: 介護支援専門員として個人を特定する番号であり、協議会の資格確認・会員管理の根幹となる
2. **重複防止**: `repairMemberCareManagerDuplicates_()` が存在するほど、重複が発生しやすい敏感なフィールド
3. **転籍・種別変換との連動**: 個人会員⇔事業所職員の転籍判定キーとして使用される（`convertStaffToIndividual_`, `convertIndividualToStaff_`）
4. **監査性**: 変更履歴が残らないまま変更されると、過去の研修参加記録や年会費履歴との整合性が崩れる可能性がある

## 検討すべき対応案

### 案A: 会員本人のみ変更可、管理者承認不要
- 自己申告で変更可能
- シンプルだが誤入力リスクあり
- 一意性チェック（既存 ACTIVE 会員との重複防止）は必須

### 案B: 変更申請→管理者承認フロー
- 会員が変更申請 → 管理者が承認後に反映
- データ整合性が高い
- 実装コストが高い

### 案C: 管理者のみ変更可（会員マイページでは読み取り専用）
- 最も保守的
- 変更依頼は管理者への問い合わせ経由
- 実装が最もシンプル

### 案D: 一意性チェック付き自己変更（暫定推奨）
- 会員本人が変更可能
- 保存時にサーバー側で一意性チェック（他 ACTIVE 会員との重複拒否）
- 変更ログを T_ログイン履歴 または 専用テーブルに記録
- 管理者は管理コンソールからの強制変更が引き続き可能

## 次担当者への指示

1. 上記4案について、協議会事務局の運用方針を確認する
2. 方針が決まったら `docs/05_AUTH_AND_ROLE_SPEC.md` に「CM番号変更ポリシー」として記録する
3. 実装は以下を起点とする:
   - **会員マイページ側**: `src/components/MemberForm.tsx` の `careManagerNumber` フィールドの編集可否制御
   - **バックエンド側**: `updateMemberSelf_()` の保存処理でCM番号変更時の一意性チェック（案D採用時）
   - **管理者コンソール側**: `updateMember_()` は現行どおり管理者権限で変更可能
