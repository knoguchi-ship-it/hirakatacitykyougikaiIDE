## 実施内容: パフォーマンス改善 Phase 1-2

### バックエンド (Code.gs)

#### B-03: warmUp() 強化
- 変更前: `CacheService.getScriptCache().get('warmup')` のみ（GAS V8 ランタイムを温めない）
- 変更後: `SpreadsheetApp.openById(DB_SPREADSHEET_ID_FIXED)` + `fetchAllDataFromDb_()` を実行
- 効果: コールドスタート時の初回リクエスト +2〜3秒を抑制。5分おきのトリガーがキャッシュも温める

#### B-01: getMemberPortalData_() 研修フィルタ最適化
- 変更前: T_研修 全件を毎回スキャン（研修数が増えるにつれ転送量が増大）
- 変更後: 申し込み済み研修ID セットを事前構築し、「申し込み済み or 今後の開催予定 (非キャンセル・未来日)」のみを返す
- 効果: 研修数が多いほど効果が大きい。DBスキャン量と JSON 転送量を削減

#### B-04: キャッシュ TTL 延長
- ALL_DATA_CACHE_TTL_SECONDS: 300 → 600秒
- ANNUAL_FEE_CACHE_TTL_SECONDS: 300 → 600秒
- 効果: キャッシュ再スキャン頻度が半減。書き込み後クリアの実装が網羅済みであるため整合性に問題なし

#### AI生成メールの GAS サーバー移行（セキュリティ + バンドル削減）
- 変更前: フロントエンドが `@google/genai` SDK を使い、API キーをビルド時に埋め込んでいた
- 変更後: `generateTrainingEmailWithAI_()` を GAS に追加。Gemini API を `UrlFetchApp` で呼び出す。API キーは `Script Properties` に `GEMINI_API_KEY` として保存
- 効果: API キーのクライアント露出を排除（セキュリティ向上）。`@google/genai` をフロントエンド依存から除去

### フロントエンド

#### B-05: openMemberDetail の不要 force reload 解消
- 変更前: 管理コンソールで会員詳細を開くたびに `loadAppData({ force: true })` を呼んでいた
- 変更後: `fullDataLoaded` かつ members state に対象会員が存在する場合はキャッシュを使い、再取得しない
- 効果: 詳細ページ表示の 1〜2 秒の待機を解消（データロード済み状態）

#### B-06: recharts → 純 SVG 実装（Dashboard.tsx）
- 変更前: recharts を使い PieChart (donut) + BarChart (horizontal) を描画
- 変更後: 外部依存なしの純 SVG + React 実装。DonutChart / HorizontalBarChart を新設
- 効果: recharts 依存を排除（Vite tree-shaking で既に効率的だったが、依存管理・ビルド時間・セキュリティリスクを低減）

#### Terser minification 追加 (vite.config.ts)
- esbuild (Vite デフォルト) から terser に変更、`passes: 2` で 2 パス最適化
- 効果: バンドルサイズ削減
