# データモデル設計書（スプレッドシートDB版）

更新日: 2026-04-06

## 1. 設計方針
- JSON列へ丸ごと保存する方式は採用しない（設定値のみJSON列を例外的に使用する）。
- スプレッドシートを「マスタ」と「トランザクションテーブル」に分割する。
- 1シート1責務、1行1レコード、主キー/外部キー列を明示する。
- 列名・マスタ名・テーブル名はすべて日本語命名とする。

## 2. マスタ一覧

### 2.1 `M_会員種別`
- 用途: 会員の区分管理
- 列: `コード`, `名称`, `表示順`, `有効フラグ`, `年会費金額`
- 初期値:
  - `INDIVIDUAL`（個人会員, 3,000円）
  - `BUSINESS`（事業所会員, 8,000円）
  - `SUPPORT`（賛助会員, 5,000円）

### 2.2 `M_会員状態`
- 用途: 会員状態の管理
- 列: `コード`, `名称`, `表示順`, `有効フラグ`
- 初期値: `ACTIVE`（有効）, `WITHDRAWAL_SCHEDULED`（退会予定）, `WITHDRAWN`（退会）

### 2.3 `M_発送方法`
- 用途: 通知媒体（メール/郵送）
- 列: `コード`, `名称`, `表示順`, `有効フラグ`

### 2.4 `M_郵送先区分`
- 用途: 定期郵送先（自宅/勤務先）
- 列: `コード`, `名称`, `表示順`, `有効フラグ`

### 2.5 `M_職員権限`
- 用途: 事業所職員の権限（3階層）
- 列: `コード`, `名称`, `表示順`, `有効フラグ`
- 初期値:
  - `REPRESENTATIVE`（代表者）: 事業所の代表。職員の追加・削除、代表者変更が可能
  - `ADMIN`（管理者）: 職員の追加・削除が可能（代表者の変更は不可）
  - `STAFF`（一般）: 自分の情報のみ閲覧・編集可能

### 2.6 `M_職員状態`
- 用途: 在籍/退職
- 列: `コード`, `名称`, `表示順`, `有効フラグ`

### 2.7 `M_システムロール`
- 用途: ログイン主体ごとのシステム権限
- 列: `コード`, `名称`, `表示順`, `有効フラグ`
- 初期値:
  - `OFFICE_ADMIN`（事務局管理者）
  - `INDIVIDUAL_MEMBER`（個人会員）
  - `BUSINESS_ADMIN`（事業所管理者）
  - `BUSINESS_MEMBER`（事業所メンバー）

### 2.8 `M_研修状態`
- 用途: 受付中/受付終了
- 列: `コード`, `名称`, `表示順`, `有効フラグ`
- 初期値: `OPEN`（受付中）, `CLOSED`（受付終了）

### 2.9 `M_申込状態`
- 用途: 申込状態（申込済/取消）
- 列: `コード`, `名称`, `表示順`, `有効フラグ`

### 2.10 `M_会費納入状態`
- 用途: 年会費の納入状態
- 列: `コード`, `名称`, `表示順`, `有効フラグ`

### 2.11 `M_申込者区分`（2026-03-12 追加）
- 用途: 研修申込者が会員か非会員かを区別する
- 列: `コード`, `名称`, `表示順`, `削除フラグ`
- 初期値: `MEMBER`（会員）, `EXTERNAL`（非会員）

### 2.12 `M_管理者権限`
- 用途: 管理画面の権限レベル管理
- 列: `コード`, `名称`, `表示順`, `有効フラグ`

#### T_会員 会員種別別フィールド制約（v131〜）

事業所会員（`会員種別コード = BUSINESS`）は法人（事業所）であり、個人属性フィールドは使用しない。

| フィールド | 個人会員 (INDIVIDUAL) | 事業所会員 (BUSINESS) | 賛助会員 (SUPPORT) |
|-----------|----------------------|----------------------|-------------------|
| 姓/名/セイ/メイ | 必須 | **ブランク（使用しない）** | 必須 |
| 介護支援専門員番号 | 必須（8桁半角数字） | **ブランク（使用しない）** | 任意（入力時は8桁半角数字） |
| 代表メールアドレス | 任意 | 事業所連絡先として使用 | 任意 |
| 勤務先名〜勤務先住所2 | 郵送先区分が `OFFICE` の場合は事業所名のみ必須。住所2は任意 | **基本情報として使用（住所2は任意）** | 郵送先区分が `OFFICE` の場合は事業所名のみ必須。住所2は任意 |
| 勤務先電話番号〜勤務先FAX | 勤務先電話番号または携帯電話番号のどちらか1つが必須。FAXは任意 | **基本情報として使用（勤務先電話番号必須、FAXは任意）** | 勤務先電話番号または携帯電話番号のどちらか1つが必須。FAXは任意 |
| 自宅郵便番号〜自宅住所2 | 郵送先区分が `HOME` の場合に使用。郵便番号・都道府県・市区町村・番地を必須、住所2は任意 | 使用しない | 郵送先区分が `HOME` の場合に使用。郵便番号・都道府県・市区町村・番地を必須、住所2は任意 |
| 発送方法コード | 使用 | **ブランク（使用しない）** | 使用 |
| 郵送先区分コード | 使用 | **ブランク（使用しない）** | 使用 |
| 事業所番号 | 使用しない | 必須 | 使用しない |

- `updateMember_()` は事業所会員保存時に上記ブランク対象フィールドを空文字で強制上書きする。
- 事業所会員の個人属性（姓名・フリガナ等）は `T_事業所職員` の各職員レコードが保持する。
- 個人会員・賛助会員は `勤務先名` が空白または `勤務なし` の場合、所属なしとして正規化する。
- 画面上の郵便番号入力 UI は 3 桁 + 4 桁の分割入力を標準とするが、`T_会員.勤務先郵便番号` / `自宅郵便番号` の保存値は `123-4567` 形式の単一文字列を維持する。
- 公開ポータルの新規入会申込では、個人会員・賛助会員・事業所職員の照合キーを `介護支援専門員番号` とする。
- 公開ポータルの事業所会員申込では、`T_会員.事業所番号` を用いて同一事業所の二重登録を拒否する。
- 申込時の「登録済み」判定は、`T_会員` では `会員状態コード != WITHDRAWN` かつ `削除フラグ = false`、`T_事業所職員` では `職員状態コード = ENROLLED`・`削除フラグ = false`・親会員が有効、を条件とする。
- 同一 `介護支援専門員番号` または `事業所番号` に複数の有効候補が存在する場合は、整合性異常として公開申込を停止し、事務局問い合わせに誘導する。

#### `費用JSON` スキーマ（実装確認済み）
**型**: `Array<{label: string, amount: number}>`
```json
[
  { "label": "会員", "amount": 0 },
  { "label": "非会員", "amount": 1000 }
]
```
- `label` は任意文字列（UI表示に使用）。
- `amount` は整数（円）。0 は無料を意味する。
- 空・null の場合は `[{label:"会員",amount:0},{label:"非会員",amount:0}]` でデフォルト補完される。
- 旧形式（数値文字列）は `[{label:"会員",amount:N}]` に変換して互換維持。

#### `項目設定JSON` スキーマ（実装確認済み）
**キーはすべて camelCase（英語）**
```json
{
  "fieldConfig": {
    "organizer": true,
    "isNonMandatory": false,
    "summary": true,
    "description": true,
    "location": true,
    "instructor": false,
    "applicationOpenDate": true,
    "applicationCloseDate": true,
    "fees": true,
    "guidePdfUrl": true
  },
  "cancelAllowed": true,
  "inquiryPerson": "事務局 田中",
  "inquiryContactType": "EMAIL",
  "inquiryContactValue": "support@example.com"
}
```
- `fieldConfig` の各フラグは `true` = 表示する、`false` = 非表示。`null` の場合はすべて `true`（全表示）。
- `inquiryPerson`・`inquiryContactValue` は `saveTraining_()` での入力バリデーション対象（必須）。
- `inquiryContactType`: `'EMAIL'` または `'PHONE'`。
- 旧形式（`fieldConfig` オブジェクトのみ / `cancelAllowed` なし）は読込時に互換変換される。

### 3.9 `T_研修申込`
- 用途: 会員・非会員を統合した申込履歴（中間テーブル）
- 主キー: `申込ID`
- 外部キー:
  - `研修ID` -> `T_研修.研修ID`
  - `申込者ID` -> `T_会員.会員ID`（`申込者区分=MEMBER`）または `T_外部申込者.外部申込者ID`（`申込者区分=EXTERNAL`）
  - `職員ID` -> `T_事業所職員.職員ID`（事業所会員職員は設定、それ以外は NULL）
- 列:
  - `申込ID` (PK)
  - `研修ID` (FK)
  - `会員ID` (互換列・NULL可) ← 既存運用との互換維持のため残置
  - `申込者区分コード` (FK -> `M_申込者区分.コード`) ← **追加（2026-03-12）**
  - `申込者ID` ← **追加（2026-03-12）**: `申込者区分=MEMBER` なら会員ID、`EXTERNAL` なら外部申込者ID
  - `職員ID` (NULL可) ← 既存。事業所会員の職員申込時のみ設定
  - `申込状態コード` (FK -> `M_申込状態.コード`)
  - `申込日時`
  - `取消日時`
  - `備考`
  - `作成日時`
  - `更新日時`
  - `削除フラグ`
- **現行実装の設計方針（移行中）**: 主設計は `申込者区分コード + 申込者ID` のポリモーフィック構成。会員申込API（`applyTraining_`）は新規レコードに `申込者区分コード='MEMBER'` / `申込者ID=会員ID` を保存する。
- **読取互換**: 申込一覧・メール送信対象生成・会員申込判定は `申込者区分コード + 申込者ID` を優先し、旧レコード（`会員ID` のみ）には後方互換フォールバックで対応する。
- **移行**: 既存レコードは `backfillApplicationApplicantIdentity_()` で `申込者区分コード=MEMBER`・`申込者ID=会員ID` を自動補完する。
- **運用方針**: 新規書込は `申込者区分コード + 申込者ID` を正とし、`会員ID` は互換列として維持する。

## 4. 実装メモ（GAS）
- `backend/Code.gs` の `setupDatabase()` で上記シートを自動生成する。
- `setupDatabase()` 実行時に、定義外シート（例: `シート1`）を自動削除する。
- マスタ参照列には入力規則（ドロップダウン）を設定し、コード値のみ入力可能にする。
- ヘッダー行は保護（警告表示）し、誤編集を防ぐ。
- `T_画面項目権限` は初期権限行を自動投入する。
- 再構築用に `rebuildDatabaseSchema()` を用意し、定義外シート削除を含めて再適用できる。
- 清掃専用に `cleanupDatabaseSheets()` を用意し、定義外シートのみ削除できる。

## 5. 認証・権限の運用ルール
- 全処理は `T_認証アカウント` を起点に認可判定する（会員/職員テーブルの直接参照のみで判定しない）。
- `ログインID` は一意制約として運用する（重複不可）。
- `パスワードハッシュ` と `パスワードソルト` のみ保存し、平文は保存しない。
- ログイン結果は `T_ログイン履歴` に記録し、失敗回数・ロック状態を更新する。
- 画面ごとの項目操作可否は `T_画面項目権限` で管理する（事業所管理者/事業所メンバー差分を含む）。

## 5.1 認証方式の運用（会員ID/PW + 管理者Google）
- 会員ログインは `ログインID + パスワード` を使用する（Googleログインは使わない）。
- 管理者ログインは **セッション認証**（`checkAdminBySession_()` / `Session.getActiveUser()`）を本番標準とする。IDトークン検証（`adminGoogleLogin_()`）は補完的に保持。詳細は `docs/05_AUTH_AND_ROLE_SPEC.md` §2 を参照。
- 管理者ログイン可否は `T_管理者Googleホワイトリスト` による許可制とする（Googleメールで照合、GoogleユーザーID は補助キー）。
- `T_認証アカウント.認証方式` は少なくとも `PASSWORD` / `GOOGLE` を保持する。
- `T_認証アカウント` に `GoogleユーザーID` / `Googleメール` を保持し、監査と突合に利用する。
- `T_ログイン履歴.認証方式` に実際のログイン方式を記録する。
- `T_認証アカウント.ログインID` は会員マイページ用の `PASSWORD` 認証識別子として扱い、管理者セッションの Google メールとは別ドメインの値として分離する。

## 5.2 管理者と会員ページの紐付け運用
- 管理者であっても `T_認証アカウント.会員ID` を必須で保持し、必ず自分の会員情報に紐づける。
- 管理者ログイン時は `管理者ページ` と `会員マイページ` の両方を表示可能にする。
- 管理者ログイン後の左メニュー表示名・会員マイページ種別は、`T_管理者Googleホワイトリスト` に紐付く会員・職員情報を優先表示する。
- `T_管理者Googleホワイトリスト` 側にも `紐付け認証ID` / `紐付け会員ID` を保持し、起動時に二重チェックする。
- 紐付け不整合（ホワイトリストと認証アカウントの会員ID不一致）はログイン拒否とする。
- 管理者セッションから会員マイページを再読込する際は `memberId/staffId` を解決キーとし、Google メールを `getMemberPortalData_()` の `loginId` 引数へ流用しない。
- `T_管理者Googleホワイトリスト` の追加・更新・削除時は、`admin_wl_v1` と `admin_auth_v1` の両キャッシュを即時無効化する。

## 5.3 退会・削除フラグ運用ルール（2026-03-09）
- `入会日` と `退会日` を会員・職員それぞれに保持する。
- 退会/退職時、原則は当年度中 `削除フラグ=false` とする（履歴保持）。
- 「中途退会の意図あり」を選択した場合のみ、更新時に即時 `削除フラグ=true` とする。
- 中途退会意図の有無に関わらず、`退会日の翌年4/1` 到達時に `削除フラグ=true` を自動適用する。
- 自動適用は API リクエスト処理の冒頭で会員・職員双方へ実行する。
- バリデーション: 退会日は入会日以降であること。退会済み会員は退会日必須。退職済み職員は退会日必須。

## 6.1 年会費管理の業務制約（2026-03-15 追加）
- `T_年会費納入履歴` は `会員ID + 対象年度` の組み合わせで重複登録しない。
- `会費納入状態コード='PAID'` の場合、`納入確認日` を必須とする。
- `会費納入状態コード='UNPAID'` の場合、`納入確認日` は空を許容する。
- 年会費管理コンソールの `前年度末退会` はドラフト専用アクションであり、`会費納入状態コード` には保存しない。
- `金額` は `M_会員種別.年会費金額` を正とし、画面入力での任意変更を認めない。
- 管理画面は対象年度の支払対象会員のみを一覧表示し、未作成レコードは画面上「未納」として扱ったうえで保存時に新規作成する。
- 会員マイページでも、現年度の年会費レコードが未作成であっても支払対象会員には当年度を `未納` として補完表示する。
- 会員マイページで未納時に表示する共通振込先は `T_システム設定.ANNUAL_FEE_TRANSFER_ACCOUNT` を正本とし、年会費レコード個別の口座情報がある場合のみそちらを優先する。
- 退会年月日が対象年度の前年度末以前の会員は、その年度の年会費支払対象に含めない。
- 年会費管理コンソールから前年度末退会を実行した場合、対象年度の年会費レコードが存在すれば論理削除し、年会費対象外と整合させる。
- 入会年月日が対象年度末より後の会員は、その年度の年会費支払対象に含めない。
- 会員マイページ向けの `annualFeeHistory` は「当年度・前年度」の連続 2 年を簡易表示の正本とする。
- 当年度または前年度の年会費レコードが未作成でも、対象会員であれば該当年度を `未納` として補完表示する。
- 管理者向け完全履歴とは分離し、旧年度実績は会員マイページの連続年度表示より優先しない。
