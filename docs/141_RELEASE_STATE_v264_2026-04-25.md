# Release State: v264

リリース日: 2026-04-25
本番バージョン: `v264`
GAS バージョン: 統合プロジェクト @266 / 会員 split @20 / 管理者 split @24

---

## 変更内容

### 1. 公開ポータル 変更・退会フロー全面刷新（OTPなし → 承認ワークフロー方式）

#### 旧フロー（v260）
CM番号入力 → OTP送信 → コード入力 → フィールド選択 → 入力 → DB即時反映

#### 新フロー（v264）
会員種別選択 → 本人確認入力 → フィールド選択（チェックボックス） → 変更内容入力（デフォルト値なし） → 申請送信（T_変更申請へ保存）→ **管理者承認後にDB反映**

### 2. 本人確認方式

| 会員種別 | 照合情報 |
|---|---|
| 個人会員 | 介護支援専門員番号（8桁）+ 氏（姓）+ 名 |
| 事業所会員 | 事業所番号 |

返信用メールアドレス: DBとは紐づかない連絡先専用。申請受付・処理結果の通知に使用。

### 3. 変更可能フィールドグループ

#### 個人会員
- 氏名セット（氏・名・フリガナ氏・フリガナ名）
- 連絡先（メールアドレス・携帯電話番号）
- 勤務先電話・FAX番号
- 勤務先住所セット（〒・都道府県・市区町村・番地・建物名）
- 自宅住所セット（〒・都道府県・市区町村・番地・建物名）
- 介護支援専門員番号 ⚠（承認時ログインIDも変更）
- 通知方法（メール/郵送）
- 郵送先区分（勤務先/自宅）

#### 事業所会員
- 事業所基本情報（名称・メール・電話・FAX）
- 事業所住所セット（〒・都道府県・市区町村・番地・建物名）
- 事業所番号 ⚠（承認時ログインIDも変更）
- 職員追加（追加可能枠数分のカードを自動表示）
- 職員除籍（姓・名・CM番号の3点照合）

### 4. セキュリティ設計

- 変更内容テキストボックスのデフォルト値なし（既存DB情報を一切表示しない）
- 申請はDB非反映。T_変更申請テーブルに PENDING として保存
- 管理者承認 → APPROVED → updateMember_() / スタッフ操作が適用
- 管理者却下 → REJECTED → DB変更なし
- 申請者・管理者双方にメール通知（MailApp.sendEmail）
- レート制限: 同一ID 15分以内5回まで

### 5. T_変更申請テーブル（新規追加）

```
申請ID, 会員ID, 会員種別コード, 申請種別コード, 申請状態コード,
申請内容JSON, 連絡先メールアドレス, 申請者表示名, 申請日時,
処理日時, 処理者メールアドレス, 処理備考, 作成日時, 更新日時, 削除フラグ
```

初回申請時に自動作成（シートが存在しない場合のフォールバック実装済み）。
正式には `npx clasp run rebuildDatabaseSchema` で明示的に作成すること。

### 6. 管理者コンソール: 変更申請管理コンソール（新規）

- サイドバーに「変更申請管理コンソール」追加（ADMIN/MASTER権限）
- 申請一覧表示（PENDING/承認済/却下済フィルター）
- 変更内容の詳細確認（フィールド変更・職員追加/除籍内容）
- 承認ボタン → DB即時反映 + 申請者へ承認通知メール
- 却下ボタン（理由必須） → DB変更なし + 申請者へ却下通知メール

### 7. バックエンド変更

| 関数 | 変更内容 |
|---|---|
| `verifyMemberIdentityForPublic_` | OTPなし本人確認（v264実装分、今回フロントエンド接続） |
| `submitPublicChangeRequest_` | T_変更申請への申請保存（自動シート作成追加） |
| `getPublicAvailableStaffSlots_` | 事業所スタッフ追加可能数取得 |
| `getAdminChangeRequests_` | 管理者: 申請一覧取得 |
| `approveAdminChangeRequest_` | 管理者: 承認+DB反映（MEMBER_UPDATE時にstaffAdd/staffRemoveも適用するよう拡張） |
| `rejectAdminChangeRequest_` | 管理者: 却下 |
| `initializeSchema_` | T_変更申請の normalizeTableColumns 追加 |

## 動作確認事項（操作者確認待ち）

1. **公開ポータル: 変更フロー**
   - 「会員登録情報を変更する」カードをクリック
   - 会員種別選択画面（個人/事業所）が表示される
   - 個人: CM番号+氏+名+返信メール入力 → 照合成功 → フィールド選択画面
   - 事業所: 事業所番号+返信メール入力 → 照合成功 → フィールド選択画面
   - チェックボックスでグループ選択 → デフォルト値なしの入力欄表示
   - 申請送信 → 返信メールアドレスに受付確認メールが届く

2. **公開ポータル: 退会フロー**
   - 「退会を申し込む」カードをクリック
   - 会員種別選択 → 本人確認入力 → 退会確認 → 申請送信

3. **管理者コンソール: 変更申請管理**
   - サイドバーに「変更申請管理コンソール」が表示される
   - 申請一覧に PENDING の申請が表示される
   - 「変更内容を確認」で申請内容が確認できる
   - 承認 → DBに反映され、申請者に承認メールが届く
   - 却下（理由入力必須） → DBは変更されず、申請者に却下メールが届く

4. **T_変更申請シート**: 初回申請後にメインDBスプレッドシートに自動作成されること

## 注意事項

- 旧OTPフロー（sendPublicOtp / verifyPublicOtp）のAPIは互換のため残存しているが、フロントエンドからは呼ばれない
- MEMBER_UPDATE承認時に staffAdd・staffRemove が changeData に含まれる場合も一括適用される
- 介護支援専門員番号・事業所番号の変更が承認されると T_認証アカウントのログインIDも同時更新される
