## 3. 今後の実装で採るべき方針

### 3.1 デプロイとビルド

- 固定 2 Deployment ID 運用を維持し、`clasp deploy` は使わない。
- `build -> push -> version -> Manage deployments -> /exec 確認 -> 実ブラウザ確認` を 1 セットで扱う。
- Vite の `build.target` は、公開ポータルが存在するため `esnext` を安易に採用しない。
- `build.target` を明示する場合は、Vite 既定 target を明示するか、サポートブラウザ方針を文書化した上で設定する。
- `dist/` / `dist-public/` から `backend/` への反映漏れを防ぐため、ビルド後コピー手順を常に確認する。

### 3.2 アクセシビリティ

- スキップリンク、ビュー遷移時フォーカス移動、`document.title` 更新、`aria-live` は標準実装として維持する。
- モーダルには visible title と `aria-labelledby` を付与する。
- ダイアログ表示時は、フォーカスがダイアログ内部へ移ることを前提に設計する。
- 追加画面を実装する際は、最低でも次を確認する。
  - キーボードのみで到達できる
  - 初回フォーカス位置が論理的である
  - スクリーンリーダー向けの状態通知がある
  - ターゲットサイズが十分である

### 3.2.1 会員マイページ表示原則

- 会員向け UI に管理者専用項目を disabled 表示で残さない。非管理者には非表示にする。
- 会員本人に必要な status は残してよいが、表現は管理用語ではなく利用者向け語彙を使う。
- 先頭カードには「現在の状態」と「次に必要な行動」だけを置き、識別子や内部管理情報を詰め込まない。
- 変更できない認証情報は、必要時だけ見る詳細セクションへ集約し、上部要約へ重複表示しない。
- ログイン ID のような固定識別子は、問い合わせや認証導線に必要な場合のみ表示し、説明文を添える。

### 3.3 個人情報保護・法令対応

- 新しい入力フォームには、利用目的通知と同意導線を最初から含める。
- プライバシーポリシーはポータル内へ複製せず、本サイト別建ての正本を参照する。ポータル側は「正本の所在」を示すにとどめる。
- 委託先管理、漏えい報告手順、プライバシーポリシーの参照方針は実装と同時に更新する。
- 「どこに保存されるか」は契約プランと設定値に依存するため、推定で断定しない。
- 委託先管理では、少なくとも次を記録する。
  - 契約プラン
  - DPA/DPAT 同意有無
  - データリージョン設定
  - 確認日
  - 確認者
  - 証跡保存場所

### 3.4 セキュリティ

- パスワードハッシュの長期方針は Apps Script 内延命ではなく、認証基盤の外出しを優先する。
- 第一候補は GCP 移行時の Firebase Auth / Identity Platform 統合。
- 中間策が必要な場合は、認証処理のみ Cloud Run / Cloud Functions に切り出す方針を優先する。
- 依存関係の追加時は `npm audit` と更新理由を記録する。

### 3.5 Apps Script / パフォーマンス

- 大きなレスポンスは chunked cache 前提で設計する。
- Apps Script の日次クォータは変動しうるため、固定値を前提にしない。
- 監視対象は次に絞る。
  - 1回の実行時間
  - CacheService 1キーサイズ
  - HTML バンドルサイズ
  - 一斉メール送信量
- TTL 変更は「体感改善」だけでなく、更新頻度・stale 許容時間・invalidations 実装有無をセットで判断する。

### 3.6 ドキュメント運用

- 状態の正本は `HANDOVER.md` と `docs/20_NEXT_INSTRUCTIONS_FOR_CLAUDECODE_2026-03-19.md` に置く。
- task 文書は実行管理用であり、完了結果を正本へ転記して閉じる。
- 変動しやすい外部仕様は、固定値を断定するより「公式リンク + 最終確認日 + この案件の判断基準」で書く。
- 文書更新時は UTF-8 表示確認を必須とする。
