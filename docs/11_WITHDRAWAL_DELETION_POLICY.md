# 退会・削除フラグ運用ルール（2026-04-22 更新）

## 対象
- 個人会員
- 事業所会員
- 賛助会員
- 事業所職員（会員配下メンバー）

## 会員状態の遷移（3状態）

```
ACTIVE（有効） → WITHDRAWAL_SCHEDULED（退会予定） → WITHDRAWN（退会済）
         ↑_________________________________↓（取消可能）
```

### 年度末退会予約方式（v104 導入）
- 会員は **会員マイページから退会申請（`withdrawSelf`）** を行うことができる。
- 退会申請時、会員状態は `WITHDRAWAL_SCHEDULED` に変更され、退会日は **当年度末（3月31日）** に自動設定される。
- `WITHDRAWAL_SCHEDULED` の間、会員は引き続きログイン可能（`T_認証アカウント.アカウント有効フラグ` は変更しない）。
- 会員は年度末到来前であれば **退会取消（`cancelWithdrawalSelf`）** を自ら行える。取消時は `ACTIVE` に戻り、退会日はクリアされる。
- **自動昇格**: `promoteScheduledWithdrawals_()` により、退会日を過ぎた `WITHDRAWAL_SCHEDULED` 会員は自動的に `WITHDRAWN` に変更される。この処理は API リクエスト処理の冒頭で実行される。

### 権限制約
- **事業所会員**: 退会申請は **代表者（REPRESENTATIVE）のみ** が実行可能。管理者・一般職員は退会申請できない。
- **個人会員/賛助会員**: 本人が直接退会申請可能。

### パスワード再認証
- 退会申請時はパスワード再入力による本人確認が必須（OWASP セキュリティガイドライン準拠）。

## ルール
1. `入会日` と `退会日` を会員・職員それぞれに保持する。
2. 退会/退職時、原則は当年度中 `削除フラグ=false` とする（履歴保持）。
3. 「中途退会の意図あり」を選択した場合のみ、更新時に即時 `削除フラグ=true` とする。
4. 中途退会意図の有無に関わらず、`退会日の翌年4/1` 到達時に `削除フラグ=true` を自動適用する。
5. 自動適用は API リクエスト処理の冒頭で会員・職員双方へ実行する。
6. データ管理コンソールの削除操作は、物理削除ではなく論理削除とする。会員・職員・認証・管理者ホワイトリストを即時無効化し、履歴テーブルは保持する。

## バリデーション
- 退会日は入会日以降であること。
- 退会済み会員は退会日必須。
- 退職済み職員は退会日必須。
- `WITHDRAWAL_SCHEDULED` の会員は退会日（年度末日）が自動設定される。

## 年会費管理コンソールからの退会処理（2026-03-30 追加）
- 管理者は年会費管理コンソールの一括保存で `前年度末退会` を実行できる。
- この操作は年会費状態の更新ではなく、会員状態を直接 `WITHDRAWN` に変更する管理操作として扱う。
- 退会日は対象年度の前年度末、退会処理日は実行日を設定する。
- 対象年度の年会費レコードが存在する場合は論理削除し、当年度の年会費対象外と整合させる。
