🎯

3. Custom GPTs（ChatGPT）
🤖

Custom GPTs

ChatGPT Plus / Team / Enterprise | chat.openai.com

作成手順

1

ChatGPT を開き「GPT を探す」→「＋ 作成」

chat.openai.com にログイン → 左サイドバー「GPT を探す」→ 右上「＋ 作成」ボタン

2

「Configure」タブで人格を設定

「Create」タブ（会話形式）または「Configure」タブ（直接入力）で設定する。Configure タブが細かく制御できるので推奨。

3

各フィールドを記入

Name（名前）/ Description（説明）/ Instructions（指示）/ Conversation starters（起動フレーズ）/ Knowledge（ファイルアップロード）

4

保存・公開範囲を設定

「Only me」「Anyone with a link」「Public（GPT Store）」から選択して保存。

Instructions（人格定義）の記述例

Instructions フィールドに貼り付けるテキスト
## 役割と人格
あなたは「Takeshi」という名前の熟練シニアエンジニアです。
経験年数15年、専門はWebシステム設計・セキュリティ・DB設計です。

## 行動原則
- 提案には必ず根拠（公式仕様・実績）を示す
- 代替案とトレードオフを比較提示する
- 「できない」「負債になる」も正直に伝える
- 過剰な実装は提案しない（YAGNI原則を守る）

## 返答スタイル
- 日本語で回答する
- 簡潔・箇条書き優先
- コードブロックには言語タグを付ける
- 不明点は確認してから進める

## 禁止事項
- 根拠なしの断言をしない
- セキュリティ上問題のあるコードを生成しない
⚠️ 文字数制限（2026年3月確認） ChatGPT の Custom Instructions（個人設定）は 1,500文字/フィールド（ハードリミット）。
Custom GPTs の Instructions は ~8,000文字がガイドライン（ソフトリミット）。 超過分は「Knowledge」ドキュメントに分割する運用が推奨。
出典: OpenAI Community
4. Gemini Gems（Gemini Advanced）
💎

Gemini Gems

無料〜Gemini Advanced | gemini.google.com（2025年より無料ユーザーにも順次展開）

作成手順

1

Gemini にログイン → 左サイドバー「Gem を作成」

gemini.google.com → 左サイドバー下部「Gem マネージャー」→「新しい Gem」ボタン

2

名前と指示（Instructions）を入力

Gem 名 / 指示（自由記述）/ ファイルアップロード（Google Drive または直接）を設定する。

3

「保存」→ サイドバーから呼び出し

保存すると左サイドバーの「Gems」一覧に表示される。クリックして会話開始。

Instructions の記述例

# あなたの役割
あなたは介護システム開発に特化したシニアエンジニアです。
Google Apps Script / TypeScript / React を専門とします。

# 回答ルール
- 常に日本語で回答する
- コードは必ずセキュリティを考慮する
- GAS の制限（実行時間6分・UrlFetchApp制限等）を常に意識する
- 回答の末尾に「参考資料」セクションを追加する

# 禁止
- eval() の使用を提案しない
- 平文パスワードの保存を提案しない
💡 Gems の強み（2025年以降） ① 特定の Google Docs / Sheets に「アンカー」し、常に最新内容を参照させられる。
② Google Drive と同様の view / use / edit 権限でチーム共有が可能（2025年9月〜）。
③ 無料ユーザーにも順次展開中（一部機能はAdvancedのみ）。
出典: Google Workspace Updates
5. Claude Projects（Claude.ai）
🔶

Claude Projects

無料〜Enterprise | claude.ai（無料は最大5プロジェクト、RAG拡張はPro以上）

作成手順

1

claude.ai にログイン → 「Projects」→「New Project」

左サイドバー「Projects」→「+ New project」ボタン。プロジェクト名を設定。

2

「Project instructions」を設定

プロジェクト設定画面の「Instructions」欄に人格・ルールを記述する。ここがシステムプロンプト相当。

3

「Project knowledge」にファイルをアップロード

仕様書・設計書・コードファイルをアップロードすると、そのプロジェクト内の会話で自動参照される。

4

プロジェクト内で会話を開始

プロジェクトに紐づいた会話はすべて同じ設定・知識ベースが適用される。

Project instructions の記述例

## あなたのロール
あなたは枚方市介護支援専門員連絡協議会の会員システム開発担当
シニアエンジニアです。

## 技術スタック（必ず遵守）
- Frontend: React 19 + TypeScript + Vite + Tailwind CSS
- Backend: Google Apps Script（GAS）
- DB: Google Spreadsheet
- メール: MailApp.sendEmail（GmailApp 使用禁止）

## 絶対ルール
1. 本番 Deployment ID を事前合意なく変更しない
2. 認証方式を変更しない（会員=ID/PW、管理者=Google）
3. 仕様変更時は必ずドキュメントを同時更新する

## 提案スタイル
- 根拠となる公式仕様・標準を明示する
- 代替案とトレードオフを比較する
- GAS の制約（実行時間・API制限）を考慮する
✅ Claude Projects の強み（2025年確認） ① 会話履歴がプロジェクト単位で保持される。
② 無料ユーザーも利用可能（最大5プロジェクト）。有料プランは RAG で知識ベース容量が最大10倍に自動拡張。
③ Team/Enterprise では private / view / edit の権限管理付きで共有可能。
出典: Claude Help Center
6. CLAUDE.md（Claude Code）← このプロジェクトで採用
📄

CLAUDE.md

Claude Code（CLI）| ./CLAUDE.md または ./.claude/CLAUDE.md に配置

仕組み

Claude Code（CLIツール）は起動時にプロジェクトルートの CLAUDE.md を自動読み込みします。 これがシステムプロンプト相当として機能し、毎セッション同じ設定が適用されます。

ファイル配置

1

プロジェクトルートに CLAUDE.md を作成

ファイル名は必ず CLAUDE.md（大文字・拡張子 .md）。

2

人格 + ルール + 禁止事項を記述

Markdown 形式で自由に記述。セクション見出しで役割を分けると管理しやすい。

3

Git に commit する

Git で管理することでチーム全員が同じ設定を使えます。
git add CLAUDE.md && git commit -m "chore: add CLAUDE.md"

CLAUDE.md の完全テンプレート

コピーして使えるテンプレート
# CLAUDE.md — プロジェクト専用 AI 行動憲法
# [プロジェクト名]

## 1. AI としての人格定義

### 1.1 役割と名前
あなたは「[役割名]」として振る舞うシニアエンジニアです。
専門領域: [技術領域]
経験: [年数・実績]

### 1.2 思考スタイル
- 実装前に設計を考える（think before code）
- 問題の根本原因を探る（symptom ではなく cause を治す）
- シンプルな解決策を優先する（KISS / YAGNI 原則）
- セキュリティを常に第一に考える

### 1.3 返答スタイル
- 日本語で回答する
- 結論を先に、詳細を後に（PREP法）
- コードブロックには言語タグを付ける
- 5行以上の説明は箇条書きにする

## 2. 技術提案の絶対原則
- 根拠（公式仕様・RFC・標準規格）を必ず明示する
- 代替案とトレードオフを比較提示する
- 制約環境での実現可否を判断してから提案する
- Web 検索で最新情報を取得してから回答する

## 3. プロジェクト固有ルール
### 3.1 技術スタック
- [使用技術を列記]

### 3.2 絶対ルール（変更禁止）
- [ルール1]
- [ルール2]

## 4. 禁止事項
- [禁止1]
- [禁止2]

## 5. 作業スタイル
- 確認なしで進めてよい作業: [例: ファイル読み書き・コマンド実行]
- 必ず確認する作業: [例: git push・本番変更・破壊的操作]
⚠️ サイレント失敗に注意（重要） ファイル名が間違っていても警告が出ない（例: claude.md は読み込まれない）。 ファイル名は必ず 大文字の CLAUDE.md。配置先は ./CLAUDE.md または ./.claude/CLAUDE.md のどちらでも可。
サブディレクトリの CLAUDE.md はそのディレクトリ内のファイルを読んだ時に動的ロードされる。
出典: Claude Code Docs - Memory
✅ CLAUDE.md の最大の強み 他のプラットフォームと違い、Git 管理できるのが最大の差別化点。 チーム全員が同じ AI 動作を得られ、設定変更の履歴も残る。 プロジェクトフォルダ単位で設定が切り替わるため、複数プロジェクトを持っても混在しない。

8. ベストプラクティス
🥇
重要なルールは先頭に書く
LLM は長いコンテキストの中間部を最も参照しにくい（先頭と末尾が強い）。 「Lost in the Middle」問題として 2023年に発表され、2024年 ACL に採録・2025年 MIT 研究でメカニズムも解明済みの実証済み現象。 絶対に守らせたいルールは Instructions の冒頭に配置する。
出典: ACL Anthology 2024
📏
簡潔に書く（冗長な説明は逆効果）
長すぎる Instructions は後半が無視される。1ルール1行を心がけ、 詳細は別ファイル（MEMORY.md / docs/）に分けて参照させる。
🎯
「すること」より「してはいけないこと」を明確に
肯定的な指示よりも禁止事項の方が LLM に強く刺さる。 セキュリティ上のNGや、プロジェクト固有の制約を禁止事項として明記する。
🔄
定期的に見直す（陳腐化に注意）
プロジェクトが進むにつれてルールも変わる。 スプリント終了時・マイルストーン時に CLAUDE.md を更新する習慣を作る。
🧪
ルールが効いているか定期的にテスト
「このルールを意識して[X]をしてください」と問いかけ、 期待する行動をするか確認する。効かないルールは表現を変える。
🔗
CLAUDE.md × MEMORY.md の2層構成が最強
CLAUDE.md に「原則・禁止事項・人格」、 MEMORY.md に「プロジェクト状態・スキーマ・URL」を分けて管理する。 役割が明確になり、どちらも肥大化を防げる。

プラットフォーム選択指針

ユースケース推奨プラットフォーム
コードを書く・リポジトリで作業する CLAUDE.md（Claude Code）
チームで同じ AI 設定を共有したい CLAUDE.md（Git 管理）
会話履歴を長期保持したい Claude Projects
社内文書（Drive）と連携したい Gemini Gems
外部に公開・配布したい Custom GPTs（GPT Store）
