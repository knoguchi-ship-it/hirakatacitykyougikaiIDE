## v238（2026-04-18 リリース）— 会員種別変換に再活性化パターン導入

### 背景・根本原因

個人⇔事業所職員の往復変換を繰り返すと、T_会員・T_事業所職員に同一介護支援専門員番号を持つ行が際限なく蓄積していた。

```
1往復ごとの変化（従来）:
  T_会員:      WITHDRAWN × N行 + ACTIVE × 1行
  T_事業所職員: LEFT × N行 + ENROLLED × 1行（or LEFT × (N+1)行）
```

さらに `assertSingleActiveAffiliationByCareManager_` が **DB 変更後** に走っていたため、
変換が「半壊状態（source WITHDRAWN だが新 staff が整合違反でエラー）」になっても
ロールバックできないという危険なアーキテクチャになっていた。

### 解決方針: **再活性化（Reactivation）パターン**

> 「人物は不変。役割だけが変わる。既存の WITHDRAWN/LEFT 行があれば再利用する」

```
再活性化後:
  T_会員:      WITHDRAWN × 1行（最初のもの）+ ACTIVE × 1行（同じ行が復活）
  T_事業所職員: LEFT × 1行（最初のもの）+ ENROLLED × 1行（同じ行が復活）
  → 往復回数によらず行数は増えない
```

### 年会費継承について

T_年会費納入履歴 の `会員ID` が memberId を参照しているため、
memberId が再利用される（再活性化）ことで支払い済み履歴が**自動継承**される。
コード変更は不要。

```
個人(A001 PAID) → STAFF変換 → A001 WITHDRAWN
同年度内に個人へ戻す → A001 REACTIVATE（ACTIVE）
→ A001の T_年会費納入履歴 PAID がそのまま有効 ✓
```

### 再活性化判定の優先順位

- `入会日` は再活性化日（today）で上書き（再入会日扱い）
- 既存 WITHDRAWN が複数ある場合: `退会日` 降順で最も新しい行を選択
- CM番号が空の場合: 再活性化は試みず、従来通り新規作成にフォールバック

### `repairMemberCareManagerDuplicates_()` の仕様

T_会員 で同一CM番号を持つ複数 ACTIVE な個人/賛助会員がある場合:
- 入会日降順で最も新しい1件を残す
- それ以外を `会員状態コード=WITHDRAWN`、`退会日=today` に更新
- 返り値: `{ repaired: number, details: [{memberId, careManagerNumber}] }`
