## 1. 背景と目的

管理コンソール（会員管理）のダッシュボード数値、事業所会員の基本情報表示、および入会日のデータ欠損に関する 3 点の改善を行う。

### 1.1 ダッシュボード年度計算

**現状**: `getAdminDashboardData_()` の「今年度入会数」「今年度退会数」が**暦年（1月〜12月）**で集計されている。

```javascript
// 現状: backend/Code.gs:2763
var currentYear = new Date().getFullYear();
// → jd.getFullYear() === currentYear で判定
```

**問題**: 日本の会計年度（4/1〜翌3/31）と一致しない。例えば 2026年3月26日時点で「今年度」は **2025年度**（2025/4/1〜2026/3/31）であるべきだが、現状は暦年 2026 で集計している。

**要件**: 年度 = 4/1〜翌3/31 として集計し、ダッシュボードカード表示にも年度を明示する。

### 1.2 事業所会員の基本情報

**現状**: MemberDetailAdmin で事業所会員（BUSINESS）にも個人属性フィールド（姓/名/セイ/メイ/介護支援専門員番号）と連絡設定を表示している。

**問題**: 事業所会員は法人（事業所）であり、個人属性は不要。事業所会員の基本情報 = 勤務先情報（事業所名・事業所番号・住所・電話等）。

**要件**:
- 事業所会員の姓/名/セイ/メイ/介護支援専門員番号は DB 上ブランクとする
- 連絡設定（発送方法コード/郵送先区分コード）も DB 上ブランクとする
- 会員編集画面（MemberDetailAdmin）でこれらのフィールドを事業所会員に対して非表示とする
- 事業所会員の「基本情報」セクション = 勤務先情報として表示する

### 1.3 入会日データの欠損補正

**現状**: 移行元ソース（★会員名簿2025年度シート V列）に **307行中280件の入会日データが存在**するが、live DB の T_会員 には **211件中18件のみ**入会日が入っている。

**根本原因**: 移行コード `buildIndividualMigrationRecord_()` が `isNewEntry`（E列 = '入会'）の行のみ入会日を取り込む設計になっていた。

```javascript
// 現状: backend/Code.gs:8870
var joinedDate = rowData.isNewEntry ? (rowData.rawJoinedDate || ...) : '';
// → isNewEntry でない行は joinedDate が常に空
```

**ソース V列の実態**（2026-03-26 検査結果）:

| 区分 | E列 | 合計 | V列あり | V列なし |
|------|------|------|---------|---------|
| 個人 | (なし) | 153 | 144 | 9 |
| 個人 | 入会 | 11 | 8 | 3 |
| 個人 | 変更 | 5 | 5 | 0 |
| 個人 | 変更届 | 1 | 1 | 0 |
| 個人 | 退会 | 1 | 1 | 0 |
| 事業所 | (なし) | 111 | 98 | 13 |
| 事業所 | 入会 | 11 | 9 | 2 |
| 事業所 | 変更 | 14 | 14 | 0 |
| **合計** | | **307** | **280** | **27** |

**要件**: ソース V列の入会日を live DB の T_会員.入会日 へ補正する。ソースに入会日がない27件はブランクのまま維持。

## 2. 変更仕様

### 2.1 要件A: ダッシュボード年度計算の修正

#### 2.1.1 年度関数の新設

`getCurrentFiscalYear_()` を backend/Code.gs に追加する。

```
ルール: 4月以降 = 当年、1月〜3月 = 前年
例: 2026-03-26 → 2025年度
例: 2026-04-01 → 2026年度
```

#### 2.1.2 年度判定ロジック

`getAdminDashboardData_()` の入会日・退会日の集計を年度ベースに修正する。

```
判定: 日付 D の年度 = D.month >= 4 ? D.year : D.year - 1
「今年度」: 日付の年度 === getCurrentFiscalYear_()
```

具体的には、入会日/退会日が以下の範囲に含まれるかで判定する:
- 年度開始: `fiscalYear/4/1`
- 年度終了: `(fiscalYear + 1)/3/31`

#### 2.1.3 ダッシュボードカード表示

| カード | 現状ラベル | 変更後ラベル |
|--------|-----------|-------------|
| 5枚目 | 今年度入会数 | {YYYY}年度 入会数 |
| 6枚目 | 今年度退会数 | {YYYY}年度 退会数 |

- `{YYYY}` はバックエンドから返却する `currentFiscalYear` 値を使用
- バックエンドレスポンスに `currentFiscalYear: number` フィールドを追加

#### 2.1.4 影響範囲

| ファイル | 変更内容 |
|----------|----------|
| `backend/Code.gs` | `getCurrentFiscalYear_()` 新設、`getAdminDashboardData_()` 年度判定修正、レスポンスに `currentFiscalYear` 追加 |
| `src/App.tsx` | ダッシュボードカード 5・6 のラベルに年度表記 |
| `src/types.ts` | `AdminDashboardData` に `currentFiscalYear` 追加 |

| フィールド | DB列名 | 方針 |

**事業所会員の「基本情報」表示内容**:

- 事業所名（既存: 勤務先情報セクションから移動）
- 事業所番号（既存: 勤務先情報セクションから移動）

**「勤務先情報」セクション（事業所会員用）**:
- 郵便番号、都道府県、市区町村、住所（既存）

#### 2.2.3 バックエンド変更（updateMember_）

- 発送方法コード/郵送先区分コード も同様にブランクを強制書き込み

#### 2.2.4 既存データ補正

### 2.3 要件C: 入会日データ補正

#### 2.3.1 補正方針

ソース（★会員名簿2025年度シート V列）から入会日を読み取り、live DB の T_会員.入会日 に書き込む。

**マッピング方法**:
1. ソースの各行から氏名（L列）+ 介護支援専門員番号（M列）を取得
2. `_MIGRATION_MAP` シートで source 行番号 → 会員ID のマッピングを参照
3. T_会員 の該当会員IDの入会日を V列の値で更新

**補正ルール**:
- ソース V列に値がある行: T_会員.入会日 を更新（既に値がある場合は上書き）
- ソース V列が空の行: T_会員.入会日 は変更しない
- T_会員 にマッピングされない行（退会者等で T_会員 に存在しない場合）: スキップ

#### 2.3.2 期待される補正結果

| 区分 | 現状 入会日あり | 補正後 入会日あり（推定） |
|------|----------------|--------------------------|
| 個人会員（179件） | 9 | 約 159〜170 |
| 事業所会員（32件） | 9 | 約 23〜32 |
| **合計（211件）** | **18** | **約 182〜202** |

> ソースの 27 件の V列ブランクのうち、一部は退会者として T_会員 に存在しないため、最終的な「入会日なし」は 27 件より少なくなる可能性がある。

#### 2.3.3 補正関数

`repairJoinedDateFromSourceJson()`（dry-run 対応）

- dry-run: マッピング結果と更新対象件数をプレビュー
- 実行: バックアップ取得後に一括更新

#### 2.3.4 影響範囲

| ファイル | 変更内容 |
|----------|----------|
| `backend/Code.gs` | `repairJoinedDateFromSourceJson()` 新設 |
